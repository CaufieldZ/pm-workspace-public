# {项目名} · Context

> 本文件由本地 Opus 在方案讨论后写入，作为后续产出的权威上下文。
>
> **分工：** ChatOpus 仅做决策辩论 / 审计 / 方案 review，不写产出物；context.md / scene-list / IMAP / PRD 等所有产出物由本地 Opus 写。ChatOpus 给出的决策建议由本地 Opus 落地到对应章节。
>
> **章节属性（静态 vs 动态）：**
> - 静态章（1/2/3/4/5/6）：描述"项目现在是什么样"。内容必须反映最新状态，旧描述被新决策覆盖时必须更新。
> - 动态章（7/9）：描述"怎么变过来的"。按日期追加，不改不删，是审计轨迹。
> - 混合章（8）：执行计划 + 阻塞项是当前状态（静态），已交付记录是历史（动态）。
>
> **核心约束：静态章的任何描述不允许与动态章的最新决策矛盾。** 新增决策后必须检查并同步静态章。
>
> **项目形态：叙事型 vs 规则系统型**（决定本模板对各章节的硬度要求）：
>
> - **叙事型**：用户旅程清晰，核心信息是「谁从哪进 → 经过 → 到哪收束」。例：社区榜单 / 直播多人连麦 / 活动报名流程。第 2 章四段式 + 第 4 章场景叙事顺序 + 第 6 章 Rule 三件套**全部强制**。
> - **规则系统型**：核心信息是 IA + 数据枚举 + 规则映射，没有典型用户主流程。例：活动中心后台 / 运营配置后台 / 数据字典管理。第 7 章决策 6 列格式 + 第 6 章 Rule 三件套（适用部分）**仍强制**；允许保留 IA 章承载页面层级 / 组件复用矩阵，允许保留枚举章承载业务线 / 品牌 / 标签等数据字典，允许在第 6 章加「数据映射表」子章承载非 if-then 规则（如活动类型 → 品牌字段映射）。
>
> **判定**：项目以「用户做完一件事」为核心 = 叙事型；以「运营配置一份规则然后系统按规则运转」为核心 = 规则系统型。两类不互斥——核心叙事型项目可带运营配置子模块（如社区榜单的后台审核），核心规则系统型项目可带局部用户旅程（如活动中心的活动报名）。混合项目按主导形态选模板，子模块按各自形态写。
>
> **老项目兼容**：本模板「项目形态区分」章节生效（2026-04-30）前已存在的 context.md 保持原结构，**不强制重写**。新增内容按新格式追加：
> - 新决策按第 7 章 6 列格式追加并加 `D{n}` ID；老决策行**不补 ID 也不补 supersedes**（`active_decisions.py` 派生时把无 ID 老决策当 Active 输出，是预期行为，不是 bug）
> - 新场景按第 4 章场景编号表追加
> - 想全量迁移到新模板的老项目，单独立 PR 做，不混入功能迭代

## 2. 产品现状 `静态`

> 四段式：前三段是叙事钩子（强制），2.4 是开放细节。读者看完前三段应该一句话能复述「谁在用什么 → 哪里卡 → 为什么现在做」。

### 2.1 现状一句话

> 谁在用什么、规模如何。一句话讲清主体、规模、当前承载方式。

例：示例活动中心日活 12K，主要承载交易激励 / 新人引导 / 节日大促三类活动，目前由市场 / 运营 / 合规三方各自配置。

### 2.2 主要冲突

> 用户痛点 / 平台痛点 / 商业目标三选 ≥ 1，每条**必须含量化数字**（百分比 / 时长 / 人数 / GMV 等）。

例：
- 用户痛点：活动入口分散在 5 个一级页面，完成率仅 18%
- 平台痛点：三方配置无统一收口，合规审查每次重复 2.5 天
- 商业目标：Q3 活动转化提升 ≥ 30% 需要先做入口收敛

### 2.3 为什么是现在做

> 时机 / 竞品动作 / 资源窗口。回答「为什么不晚一个 quarter」。

例：OKX 04 月上线活动中心 V2 后周活提升 22%；Q3 营销预算定盘前必须给方案；前端有 1 个 sprint 空档。

### 2.4 现状细节（开放）
