# PM 产出物评审框架（5 维度）

> 改造自 Claude Design critique-guide.md，替换设计维度为 PM 视角。
> 适用于：cross-check skill 对 PRD / 交互大图 / 原型的最终评审。

---

## 评分标准

### 1. 场景覆盖（Scene Coverage）

| 分数 | 标准 |
|------|------|
| 9-10 | scene-list.md 所有编号在产出物中完整呈现，含子场景、边界态、空态 |
| 7-8 | 主场景全覆盖，1-2 个子场景或边界态漏掉 |
| 5-6 | 核心主路径有，但有明显的场景漏项（如某端完全缺失）|
| 3-4 | 场景清单中 >30% 编号在产出物中找不到对应 |
| 1-2 | 基本无法对照场景清单追溯 |

**评审要点**：

- scene-list.md 编号与产出物中的 Scene 编号是否一一对应？
- 是否覆盖了失败态 / 空态 / 异常路径（不只是 happy path）？
- 后台场景是否有对应的管理端 View？
- 跨端跳转（→ 见 X-0x 格式）是否标注清楚？

---

### 2. 信息层级（Information Hierarchy）

| 分数 | 标准 |
|------|------|
| 9-10 | 主需求 → 子需求 → 规则 → 数据字段，层次清晰，一眼找到核心 |
| 7-8 | 主次关系大致清楚，个别规则混在描述里不易定位 |
| 5-6 | 能分出标题和说明，但业务规则和交互说明混放 |
| 3-4 | 信息平铺，需要逐字阅读才能找规则 |
| 1-2 | 完全没有层次，读者无法快速定位所需信息 |

**评审要点**：

- 每个场景的「核心操作」是否是视觉主角？
- 边界条件 / 错误提示是否与主流程明确区分（缩进 / 颜色 / 标签）？
- PRD 中业务规则是否独立成章节，不淹没在交互描述里？

---

### 3. 规则完备（Rule Completeness）

| 分数 | 标准 |
|------|------|
| 9-10 | 每个交互节点的条件、触发、结果、限制全部有据可查 |
| 7-8 | 主流程规则完整，1-2 处边界规则未说明 |
| 5-6 | 规则存在，但有明显的「这种情况怎么处理」没有答案 |
| 3-4 | 规则碎片化，研发需要大量 clarify |
| 1-2 | 几乎没有业务规则，只有 UI 描述 |

**评审要点**：

- 是否有未填的「待确认」「TBD」字段？
- 频控 / 白名单 / 门槛等数值是否明确？
- 状态机是否完整（每个状态 → 下一个状态的所有路径）？
- context.md 第 6 章业务规则是否在产出物中有对应呈现？

---

### 4. 端能力（Platform Capability）

| 分数 | 标准 |
|------|------|
| 9-10 | 每个交互都经过端能力校验，不可能出现「H5 画需要 TRTC」这类错误 |
| 7-8 | 大部分功能端能力正确，有 1-2 处跨端描述模糊 |
| 5-6 | 端边界有，但存在模糊区域（如「App 内 H5 页」的能力边界未说明）|
| 3-4 | 有明显的跨端能力错误，如 H5 页画了 App 原生推送功能 |
| 1-2 | 完全没有端能力意识，任何端都画了任何功能 |

**评审要点**：

- H5 是否画了需要 TRTC/WebRTC 的功能？
- OBS 主播端是否画了 TRTC 专属功能？
- 跨端跳转是否用「→ 见 D-0d」格式标注？
- 浏览器 / 原生 App / H5 的能力边界是否在产出物中有注释说明？

---

### 5. 风险（Risk & Edge Case）

| 分数 | 标准 |
|------|------|
| 9-10 | 显性和隐性风险全部识别，每个风险有处置策略（降级 / 兜底 / 告警）|
| 7-8 | 主要风险识别，处置策略 1-2 处待补充 |
| 5-6 | 提到了风险但没有策略，「待确认」多 |
| 3-4 | 只有 happy path，没有失败场景和降级方案 |
| 1-2 | 没有任何风险意识 |

**评审要点**：

- 是否包含失败态（接口超时、数据为空、权限不足）？
- 资金 / 合规场景是否有二次确认或审批节点？
- 是否有降级方案（功能降级 / 白名单限量测试）？
- 第三方依赖（支付通道、KYC 服务）是否有失败 fallback？

## 场景侧重（不同产出物评审重点不同）

| 产出物 | 最重要 | 次重要 | 可放宽 |
|--------|--------|--------|--------|
| 交互大图 | 场景覆盖、端能力 | 信息层级 | 规则完备（可 PRD 补充）|
| PRD | 规则完备、场景覆盖 | 风险 | 信息层级（结构已定）|
| 原型 | 端能力、信息层级 | 场景覆盖 | 规则完备（视觉层不承载）|
| bspec | 规则完备、风险 | 场景覆盖 | 信息层级（给研发 AI，细是对的）|
| pspec | 信息层级、端能力 | 场景覆盖 | 风险（给设计 AI）|

**总体评分**：X.X/10 [优秀(8+) / 良好(6-7.9) / 需改进(4-5.9) / 不合格(<4)]

**分项评分**：
- 场景覆盖：X/10 — [一句话说明]
- 信息层级：X/10 — [一句话说明]
- 规则完备：X/10 — [一句话说明]
- 端能力：X/10 — [一句话说明]
- 风险：X/10 — [一句话说明]

### 优点（Keep）
- [具体做得好的地方，用 PM 语言描述]

### 问题（Fix）
[按严重程度排序]

**1. [问题名称]** — ⚠️ 致命 / ⚡ 重要 / 💡 优化
- 当前：[描述现状]
- 问题：[为什么是问题]
- 修复：[具体操作，含文件位置 / 行号]

### 快速修复清单（Quick Wins）
如果只有 30 分钟，优先做这 3 件事：
- [ ] [最有影响力的修复]
- [ ] [第二重要的修复]
- [ ] [第三重要的修复]
