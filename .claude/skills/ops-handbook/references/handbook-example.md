# 运营手册抽象示例

> 基线：[活动中心运营操作手册 v3](../../../../projects/growth/activity-center/deliverables/活动中心_运营操作手册_v3.docx)（13 章实战版）。本文抽出关键模式给新手册参照，**不是抄章节标题**，而是看「为什么这么分章」。

---

## 1. 章节如何按业务复杂度展开

简单后台（< 5 个核心操作）→ 6 章基础骨架够用。

复杂后台（如活动中心 13 章）→ 把核心模块拆成独立 H1，避免 H2 嵌套到 4 级。活动中心拆法：

| H1 | 业务含义 | 为什么独立成章 |
|----|---------|--------------|
| 1. 系统是什么 | 三方关系 + C 端曝光位 | 模块概览（让新人 1 分钟懂这是干啥的） |
| 2. 你的权限边界 | 三档运营权限对照 | 权限矩阵（不写清楚导致越权 / 客诉） |
| 3. 用户那边看到啥 | C 端 4 个曝光位 | 让运营理解「我配的东西用户怎么看到」（核心同理心） |
| 4. 自动同步活动从哪来 | 同步流程 + 字段映射 | 同步逻辑复杂（10 类自动 + 8 类手动），独立成章 |
| 5. 活动列表页操作 | 筛选 + 表头 + 一级编辑 | 80% 的运营时间在列表页，必须详细 |
| 6. 活动编辑页 · 基础信息 | 必填字段 + 已删字段 | 编辑页是核心，按 Tab 拆 H1 |
| 7. 活动编辑页 · 多语种配置 | 9 种语言 + 兜底规则 | 多语种规则比基础信息复杂，独立 |
| 8. 上线流转 | 状态流转 + 自检清单 | 上下线是高风险动作 |
| 9. 专题管理 | 6 个内置专题 + 三维 AND 筛选 | 专题筛选规则是活动中心最难的部分（v2 没讲清导致客诉，v3 拆成 10 个 H2） |
| 10. 排序与曝光控制 | 权重公式 + 推荐专题特殊规则 | 排序逻辑独立（运营调权重最频繁） |
| 11. 活动日历 | 新位置 + 三件事要知道 | 周边小工具 |
| 12. 字段枚举速查 | 业务线 / 品牌 / 标签 | 工具人速查 |
| 13. 高频踩坑 FAQ | 10 条 Q&A | 群里被问 ≥ 3 次的 |

**原则**：**复杂度集中在哪里，就在那里拆 H1**。专题管理在活动中心是最复杂的（10 个 H2 子节），所以独立。同样逻辑用在你的项目：哪个模块运营问题最多，那个模块就拆 H1。

## 2. 操作步骤的「三列范式」

活动中心 v3 「9.7 三个具体例子手把手算一遍」是好范本：

```
例子 1：BTC 巅峰赛 II
├── 活动属性    业务线 = 合约 / 品牌 = 巅峰赛 / 标签 = 热门
├── 命中规则    专题「巅峰赛」筛选条件：业务线 ∈ {合约,现货} AND 品牌 = 巅峰赛
└── 三维计算    业务线: 合约 ∈ {合约,现货} ✓
                品牌:   巅峰赛 = 巅峰赛 ✓
                标签:   未限制 → 自动通过 ✓
              → 命中 ✓ 落入「巅峰赛」Tab
```

**为什么是好范本**：

1. 给具体输入（活动属性），不抽象描述
2. 列规则原文（让运营对照看）
3. 三维分别算一遍（`✓ / ✗`）+ 最终结论
4. 故意挑边界场景（「未限制」自动通过是新人最容易错的）

复用到其他项目：选 3 个**典型 + 易错**的具体案例，按这个范式手把手算。

## 3. FAQ 选题的好范本

活动中心 v3 的 10 条 FAQ 全部来自实际客诉：

| Q | 实际触发场景 |
|---|------------|
| Q1 同步进来的活动直接就能在 C 端看到吗？ | 业务运营误以为同步=上线，配完就走 |
| Q4 越南语没配，越南用户看不到活动怎么办？ | 兜底规则没讲清楚，运营漏配越南语 |
| Q5 活动配完为什么命中预览全是灰底？ | 三维 AND 算错（专题筛选规则没读懂） |
| Q9 业务线运营改不动权重和热门？ | 权限矩阵没看 → 越权 |

**反例**（不要写）：
- ❌ 「Q：什么是活动中心？」（应在第 1 章模块概览）
- ❌ 「Q：我应该如何使用？」（套话，没具体场景）
- ❌ 「Q：上线后怎么修改？」（应在「上线流转」章）

**判定**：FAQ 必须是「**主流程文档讲清楚后还会被问的边界 / 例外问题**」，主流程没讲的不要靠 FAQ 兜。

## 4. 字段定义的「枚举穷举 + C 端文案对照」

活动中心 v3 「12.2 活动品牌（15 个，可多选）」结构：

| brand_id | 中文名 | C 端展示 | 适用业务线 |
|----------|-------|---------|----------|
| launchpool | Launchpool | 卡片左上角金色徽章 | 现货 / 理财 |
| peak_battle | 巅峰赛 | 卡片右下角红色标签 | 合约 |
| recharge_race | 充值赛 | 卡片角标「充」字 | 现货 / 充值 |
| ... | ... | ... | ... |

**为什么这样写**：

- 字段名（snake_case）让运营在 CMS 里照着搜
- 中文名让运营之间口头沟通
- C 端展示让运营知道选了之后用户看到什么样
- 适用业务线避免「合约活动选了 launchpool」这种错配

**禁止简化为**：「活动品牌：15 个枚举值」（运营搜不到、对不上）。

## 5. 变更记录写法

v3 的变更记录写法（节选）：

```
v3.0（2026-04-29）相对 v2.0（2026-03-15）

新增内容：
├── 第 9 章 专题管理：新增「9.4 三维 AND 怎么算」+「9.7 三个具体例子手把手算」
│   原因：v2 把三维 AND 一句话带过，运营群本月共问 17 次，3 起客诉
├── 第 13 章 FAQ：新增 Q5 / Q9 / Q10
│   原因：本月运营群高频问题
└── 第 12 章 字段枚举：业务线从 10 个扩到 12 个，活动品牌从 13 扩到 15
    原因：DEX / NFT 业务线上线，新增对应品牌

修改内容：
├── 第 7 章 多语种：兜底规则从 2 段改为 3 段（增加「越南语缺失走英语兜底」）
└── 全文：标点 / 字体三件套对齐 PRD 设计规范

删除内容：
└── 第 6 章 已删字段：补充说明 v4.5 删的「优先级」「分类」等 5 个字段
    原因：避免老同事按记忆找
```

写变更记录的关键：**写「为什么改」**（用户问得多 / 客诉 / 业务上线），不只写「改了什么」。
