## 1. 模块概览

回答 3 个问题：

1. **这个系统是什么**：1-2 句话定义。如「活动中心是 HTX 全站活动的统一管理 + 投放系统，运营在 MGT 后台配置，C 端展示在首页 / App / Web」。
2. **三方关系**：业务方 / 运营 / 用户的关系图（可用 mermaid 或 2 列表格）。
3. **C 端曝光位**：列出运营配置的活动会出现在哪几个位置（首页 banner / 活动中心 Tab / 推送等）。

文风：动词开头的描述（「展示在 ...」「触发自 ...」），不要堆术语。

明确写「业务线内」「全部」「✗」三档，不要笼统写「有权限」/「无权限」。

## 3. 核心操作流程

按用户旅程顺序展开（不是 CMS 页面顺序），即 **运营在一天工作里会顺序碰到哪些操作**：

1. 看新活动从哪来（同步流程）
2. 在列表页找到活动
3. 编辑活动基础信息
4. 配多语种
5. 上线流转
6. 配专题（决定活动落哪个 Tab）
7. 配排序权重

每个操作用 `step_table` 三列：

| # | 动作 | 看到 / 校验 | 异常态怎么办 |
|---|------|------------|------------|
| 1 | 进入「活动列表」→ 点击「新建活动」 | 跳转新建页，必填项标红星 | 没有「新建」按钮 → 你是业务运营，只能编辑同步进来的，找总运营建 |
| 2 | 填活动名称（中文 + 至少 1 种外语） | 校验：≥ 4 字符，不允许 emoji | 报「名称重复」→ 改名或找原活动负责人合并 |
| ... | ... | ... | ... |

**截图规范**：每步骤至少 1 张 CMS 后台截图（用 `[step-1]` `[step-2]` 占位符，由 `handbook_screenshots.py` 回填）。

**禁止**：步骤里写代码字段含义（如「调用 `POST /api/activity` 接口」）—— 那是 PRD / API 文档的事。

## 4. 字段定义

`field_table(doc, fields=[
    ('business_line', '业务线', '枚举', '12 个，可多选', '现货 / 合约 / 理财 / Earn / ...'),
    ('brand_id', '活动品牌', '枚举', '15 个，可多选', 'Launchpool / 巅峰赛 / 充值赛 / ...'),
    ('display_tag', '外显标签', '枚举', '11 个，单选', '热门 / 新人 / 限时 / ...'),
])`

**允许 snake_case**：运营要照着字段名找配置项，写 `business_line` 比写「业务线字段」更准确。

枚举必须穷举（不写「等」「若干」），每个枚举值给出**对应 C 端展示文案**（避免运营误选）。

## 5. 故障排查（FAQ + 错误码）

### FAQ 选题原则

只写 **PM / 运营在群里被问超过 3 次的问题**，不要凭空生成「常见问题」。

每条用 `faq_block`：

```
Q1：同步进来的活动直接就能在 C 端看到吗？
A：不能。同步只拿基础信息，外显标签 / 多语种文案 / 跳转链接需要运营手动补，
   补完点上线才会展示。看不到优先查 3 项：① 状态是不是「已上线」② 多语种
   有没有配 ③ 命中预览是不是灰底（未命中专题）。
```

### 错误码索引（如有）

| 错误码 | 触发场景 | 修复 |
|--------|---------|------|
| 4001 | 必填字段为空 | 检查表单标红项 |
| 4003 | 多语种配置不完整 | 补齐至少 1 种外语 |
| 5002 | 跳转链接格式错误 | 外链需带 `https://`；内链用 `htx://` 协议 |

## 文风纪律

| 维度 | ⛔ | ✅ |
|------|-----|-----|
| 主语 | 「系统会自动 ...」 | 「点击保存后，看到 ...」（直接告诉运营该看什么） |
| 决策痕迹 | 「v2 决定做」「与 PM 讨论后」 | 直接写当前操作，不带历史痕迹 |
| 模糊词 | 「一般来说」「通常」 | 「12 个业务线」「8 类自动同步」具体数字 |
| 半角标点 | 中文旁 `,:()` | 全角 `，：（）`（与 PRD 同 hook 拦截） |
| 圈数字 | ①②③ | 1./2./3. |

snake_case 字段名 / 像素值 / hex 色 **不限制**（运营要照着 CMS 字段配置）。
