# 场景清单 Skill（Scene List）

## 作用

场景清单是所有复杂链路的**第一步**，确定三件事：
1. **View 划分** — 产品拆为几个独立端/视角
2. **场景编号** — 每个场景的唯一编号（确定后不可改动）
3. **优先级** — P0 / P1 / P2 / 后续

后续所有产出物（交互大图、原型、PRD、需求框架）必须复用此编号和 View 划分。

## 输出格式

Markdown 表格，输出为 projects/{项目名}/scene-list.md 文件，方便后续交互大图和PRD引用。

## 生成步骤

### Step 0：需求发散（新需求 / 模糊需求时执行）

**触发条件**：用户只说了一句话（如「做个活动中心」「优化留存」「做个新功能」），或 context.md 中没有明确的问题定义和目标指标。

**跳过条件**（满足任一即自动跳过，不需要 PM 说）：
- context.md 已有完整目标/指标/场景描述（用 `grep -i "目标\|指标\|成功标准"` 确认）
- PM 说「直接梳理场景」「跳过澄清」「快速模式」
- 需求是已有功能的小改动，规模 ≤ 1 个场景

---

**0.1 确认问题定义**（逐个问，等回答再问下一个，不得合并）

问 1：**这个需求要解决谁的什么痛点？**
> 期望：一个具体场景，如「用户不知道有多少可用活动，经常错过」，不接受「优化体验」

问 2：**对应的业务目标是什么？**
> 期望：具体指标，如「提升活动参与率 15%」「日均参与活动 UV 增加 2 万」

问 3：**上线后用什么数据判断成功？**
> 期望：1-2 个核心指标 + 时间窗口，如「上线 4 周后，周活跃参与率从 8% 升到 12%」

---

**0.2 提出 2-3 个实现方向**

基于以上回答，给出 2-3 个不同路径，每个说明：
- **核心差异**：各路径的本质区别（不只是功能多少）
- **涉及范围**：哪些端/模块，大约几个场景
- **关键假设**：这个方向成立需要哪个假设为真
- **你的推荐**：明确说推荐哪个，理由是什么

等 PM 确认方向后，进入 Step 1。

---

### Step 1：收集信息

向用户确认：
1. 产品/项目名称
2. 涉及哪些端：App / Web / 后台 CMS / 后端 / H5 / 其他
3. 每个端有哪些核心页面/功能模块
4. 是否有跨端交互（如 App 操作 → Web 后台审核）
5. 哪些是必须做的（P0），哪些是增强（P1），哪些后续迭代

### Step 2：确定 View 划分

按**独立的产品端/视角**拆 View：

```
View 1 · [前台名称]（Web + App）
View 2 · [专区名称]（Web + App）— 如果有独立子产品
View 3 · [后台名称]（Web）
```

规则：
- 同一端但功能完全独立的模块可拆为不同 View（如用户前台 vs 运营专区）
- 前后台必须分 View
- View 名称要具体，不用「View 1」「View 2」这种抽象命名

### Step 3：编号并输出表格

**每个 View 一张表**，格式固定：

```markdown
## View 1 · [View 名称]

| 编号 | 场景 | 模块 | 设备 | P |
|------|------|------|------|---|
| A    | [场景名] | [所属模块] | 📱phone | P0 |
| B-1  | [场景名] | [所属模块] | 📱phone | P0 |
| B-2  | [场景名] | [所属模块] | 🖥web | P0 |
| B-3  | [场景名] | [所属模块] | 🖥web | P1 |
```

### 编号规则

| 类型 | 格式 | 示例 | 说明 |
|------|------|------|------|
| 主场景 | 大写字母 | `A` `B` `C` | 独立的页面/流程入口 |
| 子场景 | 字母-数字 | `B-1` `B-2` | 同一主场景下的子页面/子状态 |
| 后台场景 | M-数字 | `M-1` `M-2` | MGT/CMS 后台页面 |
| 功能前缀 | 缩写-数字 | `F-0` `E-1` | 特殊功能模块（F=Futures, E=Entry 等）|
| 入口子场景 | 字母-数字+字母 | `F-0a` `F-0b` | 同一场景多条独立入口路径 |

**设备列**：每个场景标注 `📱phone`（App 端）或 `🖥web`（Web 前台/后台/H5），同一 View 下可混合。交互大图的骨架脚本直接读取此列决定设备壳类型。

规则：
- 编号按 View 内顺序递增，不跳号
- 同一 View 内不同模块可用不同字母段（如 A-C 是核心模块，D-F 是扩展模块）
- **编号确认后不可改动**——新增场景在末尾追加，不插入

### Step 4：汇总统计

表格末尾附统计：

```
**统计**：共 X 个场景 · P0 × N · P1 × N · P2 × N
**涉及端**：App + Web + CMS 后台
```

### Step 5：等用户确认

输出后明确询问：
> 场景清单确认后编号将锁定，后续交互大图/原型/PRD 全部复用。请确认或提出修改。

用户确认后才进链路下一步。

## 与 requirement-framework 的关系

- **场景清单**：粗粒度，定义「有哪些场景」+ 编号 + 优先级
- **需求框架**（requirement-framework skill）：细粒度，在场景清单基础上拆出每条具体需求

场景清单的编号 = 需求框架的模块编号。两者编号体系一致。

## 自检

- [ ] 每个 View 有独立表格
- [ ] 编号连续无跳号
- [ ] 每个场景有且仅有一个优先级
- [ ] View 名称具体（不是 View 1/2/3）
- [ ] 每个场景有设备标识（📱phone / 🖥web）
- [ ] 统计数字 = 实际场景数
