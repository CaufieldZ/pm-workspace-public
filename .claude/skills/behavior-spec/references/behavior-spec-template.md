## 填充规范

### 数值处理

- PRD 中的所有数值（阈值、限制、计算公式）**原样保留**
- 不省略为「若干」「多个」「一定数量」
- 不转换精度（PRD 写「最多 5 个」就写「最多 5 个」，不写「上限 5」）

### 状态描述

- 状态名从 PRD / scene-list.md 继承，不自创
- 状态流转用表格：From → To → 触发条件 → 系统行为
- 每个状态变更必须说明「什么条件下」触发

### 异常覆盖

- 每个用户操作至少覆盖一个异常场景
- 异常场景必须包含：触发条件 + 系统响应 + 用户感知
- 常见异常类型（至少考虑）：
  - 条件不满足（未登录、资格不符、超出限制）
  - 数据异常（不存在、已过期、被删除）
  - 并发冲突（重复操作、库存不足）
  - 网络异常（超时、断连）

### 术语表

- 文档内首次出现的业务术语必须在术语表中有定义
- 定义用业务语言，不用技术术语
- 来源优先级：scene-list.md → context.md → PRD 内定义

### 跨场景交互

- 仅在存在场景间数据依赖时填写
- 无跨场景交互时可标注「本模块无跨场景交互」后跳过
- 数据流描述的是业务含义，不是技术传输方式

## 切分多份文档时的规范

当按模块拆分为多份独立文档时，每份必须：

1. **独立的术语表**：只包含本模块用到的术语（全局术语也需包含）
2. **独立的异常覆盖**：不依赖其他文档的异常描述
3. **跨文档引用用自然语言**：如「用户完成报名（见「报名模块」行为规格 Scene A）后，可在本模块查看达标进度」
4. **各自通过自检**：每份文档独立执行自检清单

文件命名：`bspec-{项目简称}-{模块名}-v{N}.md`
