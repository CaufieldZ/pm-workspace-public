# 竞品分析 Skill（Competitor Analysis）

## 定位

独立产出物，直接给老板/业务方看，不一定接后续需求链路。
分析完后截图和结论自动沉淀到 `references/competitors/{平台名}/`。

## 核心原则

> 你的任务**不是罗列竞品有什么功能**，而是：
> - **拆解设计**：功能怎么设计的、交互怎么走的、信息怎么组织的
> - **分析意图**：为什么这么设计、服务什么业务目标、解决什么用户问题
> - **评价优劣**：设计好在哪、差在哪、用户实际体验如何
> - **判断壁垒**：哪些能力本质上不可复制（路径依赖/网络效应/数据壁垒）
> - **落地追踪**：可借鉴的部分，我方 目前做到哪了

### 对比锚点（强制）

**所有分析必须包含三角对比：分析对象 vs Binance vs 我方。**
- Binance 是行业基准线，永远作为参照
- 我方 是我方现状，永远作为落地锚点
- 即使只分析一个竞品，也必须拉上 Binance 和 我方 做横向对比

### 反思维惯性（强制开启）

每次分析前先自检这 5 条，违反任何一条 = 分析失败：
1. **反「功能堆砌」迷信** — 不因功能丰富而高估竞争力
2. **反「用户量即正义」** — 不因 DAU 大而误判护城河
3. **反「大厂都这么做」** — 团队规模/技术栈/用户结构/资源差异巨大，直接抄 = 翻车
4. **反「技术万能论」** — 算法推荐 ≠ 用户留存，技术是手段不是目的
5. **反「行业报告权威论」** — 行业报告是平均水平，不是你的具体场景

### 校验标准

- 只列功能有无没拆设计细节 = **失败**
- 没分析「为什么这么设计」 = **失败**
- 没评价设计优缺点 = **失败**
- 没考虑我方约束 = **失败**
- 可借鉴点没标 我方 进度 = **失败**
- 正向结论多于反向结论 = **可疑**
- 没有 Binance 和 我方 的三角对比 = **失败**

## 两种档位

### 快速版

适合：快速了解某个功能模块、会议前快速准备、日常竞品跟踪
产出：5-8 页精炼分析

**分析维度**：
1. **功能设计拆解**（不是有/无矩阵，而是逐功能分析设计方案 + 优缺点）
2. **三角对比速览**（竞品 vs Binance vs 我方，关键差异可视化）
3. **UI/交互模式速览**（关键截图 + 设计意图分析 + 优缺点）
4. **可借鉴点 + 我方 进度**（每条标注 我方 状态）
5. **要避的坑**（直接抄会翻车的点）
6. 一句话结论 + 行动建议

### 深度版（完整 CRBE）

适合：立项前深度调研、战略级竞品对标、汇报用
产出：12-18 页完整报告

**CRBE 分析框架**：
1. **Capability（能力拆解）** — 逐项标注：✅可复制 / ❌不可复制 / ⚠️路径红利 / 💎差异化机会
2. **Route（路径分析）** — 竞品怎么走到今天的，路径依赖是什么，我们能不能走同一条路
3. **Barrier（壁垒识别）** — 网络效应/数据壁垒/品牌认知/生态锁定，哪些我们翻不过去
4. **Efficiency（效率对比）** — 同一任务竞品几步完成/我们几步，转化漏斗差异在哪

**分析维度（全量）**：
1. 功能设计深度拆解（逐功能：设计方案 + 交互流程 + 设计意图 + 优缺点）
2. 三角对比矩阵（竞品 vs Binance vs 我方，维度粒度到设计细节层）
3. 用户流程对比（同一任务各家几步完成，截图标注关键差异）
4. UI/交互模式分析（信息架构/信息密度/操作路径/设计语言 + 每个点的优缺点）
5. 数据/指标对比（TVL/参与人数/转化漏斗 — 有多少写多少，没有的标「未公开」）
6. 商业模式差异（费率/激励机制/分润/获客成本）
7. CRBE 四维分析（核心章节）
8. 反向结论：🚫 不该学的 3 件事 / 🧩 可借鉴的底层逻辑 / 💡 差异化机会
9. 翻车预警（产品/用户/技术/运营/成本 5 个层面）
10. 行动建议 + 我方 借鉴进度追踪
11. **Hook 模型诊断（留存壁垒评估）** — 分析竞品是否建立了使用习惯循环

> 习惯深度越高 = 切换成本越高 = 护城河越真实。

| Hook 阶段 | 竞品如何设计 | 我方 当前状态 | 差距评估 |
|----------|------------|------------|---------|
| **外部触发** | 哪些 push/提醒/邮件把用户拉回来？节奏是什么？ | | |
| **内部触发** | 用户什么情绪下主动打开？（FOMO/贪婪/焦虑/习惯） | | |
| **行动** | 从打开到完成核心操作最少几步？有无主动降低摩擦的设计？ | | |
| **可变奖励** | 用户每次打开能获得什么不确定的收益/惊喜/信息？ | | |
| **投入** | 用户沉淀了什么（资产/记录/关注/设置）使切换成本上升？ | | |

**Hook 强度结论**：\_\_/10（7+ 分不靠功能竞争，靠摩擦成本；<5 分容易被复制）

## 功能设计拆解规范（核心章节，不可省略）

> 这是最重要的部分。每个功能都必须按此结构拆解，禁止只写「有/无」。

### 单个功能的拆解模板

```
### {功能名称}

**设计方案**：（交互流程、页面结构、信息架构、关键截图引用）
**设计意图**：（为什么这么设计、服务什么业务目标、解决什么用户问题）
**用户雇佣此功能做什么（JTBD 透镜）**：
| 维度 | {竞品} | 我方 |
|------|--------|-----|
| **功能性目标** | 用户用它完成什么任务（如：快速找到高收益质押产品） | |
| **情感性目标** | 用户希望感受到什么（如：对收益有把握、不焦虑） | |
| **社交性目标** | 用户在圈内想呈现什么形象（如：懂 DeFi 的老手） | |
| **非竞品替代** | 用户不用此功能时转而用什么（如：Discord 问社区、Excel 手算） | |
| **切换阻力** | 已在竞品的用户为什么不迁移到 我方（资产/习惯/信任） | |
**优点**：（设计好在哪，用户体验好在哪，数据表现如果可知）
**缺点/局限**：（设计差在哪，什么场景下体验不好，有什么遗漏）
**三角对比**：
| 维度 | {竞品} | Binance | 我方 |
|------|--------|---------|-----|
| 入口位置 | ... | ... | ... |
| 交互步骤 | ... | ... | ... |
| 信息密度 | ... | ... | ... |
| 核心差异 | ... | ... | ... |

**借鉴价值**：（能不能学、学什么、直接抄的翻车风险）
**我方 进度**：🟢/🟡/🔵/🔴/⚫ + 具体说明
```

## 我方 借鉴进度追踪（每条可借鉴结论必须附）

| 状态 | 标记 | 说明 |
|------|------|------|
| 已上线 | 🟢 | 附上线版本/日期，可评估效果 |
| 开发中 | 🟡 | 附预计上线时间、当前阶段 |
| 已排期 | 🔵 | 在 Roadmap 中但未开始开发 |
| 未开始 | 🔴 | 本次分析后可考虑是否立项 |
| 评估后放弃 | ⚫ | 附放弃原因（资源/ROI/技术限制） |

格式示例：
```
3. **可借鉴：OKX 的跟单信号卡片设计** — 🟡 开发中（预计 Q2 上线，前端已完成，后端联调中）
4. **可借鉴：Binance 的策略广场社区运营** — 🔴 未开始（建议 Q3 排期）
5. **可借鉴：Bybit 的 AI 推荐策略** — ⚫ 评估后放弃（推荐算法需 6 个月+ 数据积累，当前 DAU 不支撑）
```

## 输出格式

### Markdown（默认）

直接在对话中输出，方便贴飞书/钉钉。按 `references/output-structure.md` 结构组织。

### PPT（用户要求时）

使用 ppt skill（HTML 多 Tab 文档）生成。详见 `references/ppt-spec.md`（仅 PPT 输出时读取）。

## 执行步骤

### Step 1：收集信息

向用户确认（缺什么问什么，不编造）：

1. **分析对象**：哪些竞品 × 哪个功能模块
2. **档位选择**：快速版 or 深度版（用户不说默认问一次）
3. **竞品分层**（Claude 建议，用户确认）：
   - Tier 1 直接竞品（同赛道、同用户群）
   - Tier 2 间接竞品（不同形态满足相同需求）
   - Tier 3 跨界参考（不同行业可借鉴逻辑）
4. **输出格式**：Markdown or PPT
5. **素材**：用户有截图/URL/体验录屏直接丢过来
6. **我方 当前进度**：相关功能 我方 做到哪了（用户口述即可，Claude 整理）

### Step 2：读取我方约束

如有根目录 `profile.md`，读取获取团队能力、资源上限、用户基础等约束。
没有 `profile.md` 则从 context.md 中提取，或直接跳过此步。

### Step 3：执行分析

**快速版**：
- 逐功能拆解设计（不是有无矩阵）
- 每个功能做三角对比（竞品 vs Binance vs 我方）
- 可借鉴点标 我方 进度
- 产出精炼结论

**深度版**：
- 按 CRBE 框架逐维度拆解
- 每个功能深度拆解设计方案 + 意图 + 优缺点
- 全量三角对比
- 可借鉴点标 我方 进度 + 建议排期

分析过程中强制执行：
- 每个功能必须拆「怎么设计的」+「为什么」+「好不好」，不能只说「有这个功能」
- 每个「可借鉴」必须附一个「直接抄会翻车的原因」
- 每个「可借鉴」必须附 我方 当前进度状态
- 数据没有来源的标「未公开，需验证」
- 我方约束必须贯穿全文
- Binance 和 我方 对比必须贯穿全文

### Step 4：产出 & 沉淀

1. 按选定格式输出（MD 在对话中 / PPT 生成 .pptx 文件）
2. 竞品截图存入 `references/competitors/{平台名}/{功能模块}/`
3. 分析报告存入 `projects/{项目名}/deliverables/`（如果有关联项目）
   或 `references/competitors/` 下（独立分析时）
4. **借鉴进度追踪**部分内容同步更新到关联项目的 `context.md`（如有）

### Step 5：PPT 专属——QA

生成 PPT 后必须执行：
1. 转 PDF → 转图片 → 视觉检查
2. 检查深色底上的文字可读性（对比度够不够）
3. 截图是否清晰、是否被裁切
4. 三角对比表格内容没被截断
5. 修复问题后重新验证
