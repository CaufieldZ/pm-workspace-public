# MRD 评分标准 v2

> 版本：v2 | 更新日期：2026-04-29
> v2 变更：新增 B-7 经济模型（条件性）、B-6 扩展 Go/No-Go 机制，总分从 90 升至 100 分

## 总分结构

| 模块 | 项目数 | 说明 | 模块满分 |
|:--|:--|:--|:--|
| 基础项 | 7 项 | B-1 ~ B-6 各 10 分，B-6 目标+决策机制 15 分，B-7 经济模型 10 分 | 70 |
| 加分项 | 3 项 | 每项 10 分 | 30 |
| **合计** | | | **100** |

> **v2 设计原则：** 有奖励成本 / 推广投入的 MRD 必须说明 ROI 路径；有量化目标的 MRD 必须给出止损阈值——「成功就继续」不等于有决策标准。

---

## 基础项评分细则（7 项，共 70 分）

### B-1 需求背景（10 分）

| 子维度 | 满分 | 评分标准 |
|:--|:--|:--|
| 业务背景清晰度 | 4 | 有具体业务场景 + 触发时机 = 4；模糊描述 = 2；缺失 = 0 |
| 需求来源说明 | 3 | 有数据 / 事件驱动 = 3；笼统说「用户反馈」= 1；缺失 = 0 |
| 目标用户画像 | 3 | 有角色 + 使用场景 = 3；仅提用户群体 = 1；缺失 = 0 |

---

### B-2 问题现状 + 数据分析（10 分）

| 子维度 | 满分 | 评分标准 |
|:--|:--|:--|
| 问题描述具体性 | 3 | 有具体场景 + 数量级 = 3；描述模糊 = 1；缺失 = 0 |
| 数据支撑充分性 | 4 | 有真实数据（埋点 / 调研）= 4；模糊估算 = 2；无数据 = 0 |
| 根因分析 | 3 | 分析到根因 = 3；停留在现象 = 1；无分析 = 0 |

---

### B-3 用户痛点（10 分）

| 子维度 | 满分 | 评分标准 |
|:--|:--|:--|
| 痛点真实性 | 4 | 有用研 / 访谈 / 行为数据支撑 = 4；基于推断 = 2；无依据 = 0 |
| 痛点可量化 | 3 | 有量化指标（频率 / 影响人数 / 流失率）= 3；定性描述 = 1；无 = 0 |
| 用户覆盖度 | 3 | 覆盖核心用户群 = 3；只覆盖部分 = 1；无 = 0 |

---

### B-4 竞品分析（10 分）

| 子维度 | 满分 | 评分标准 |
|:--|:--|:--|
| 竞品覆盖完整 | 4 | ≥ 3 个主要竞品 = 4；1–2 个 = 2；无 = 0 |
| 差异化分析 | 4 | 有「我们的差异化结论」= 4；仅列功能对比表 = 2；无 = 0 |
| 市场格局判断 | 2 | 有格局判断（领先 / 跟随 / 差异化）= 2；无 = 0 |

---

### B-5 解决方案（10 分）⚠️ 关键项

| 子维度 | 满分 | 评分标准 |
|:--|:--|:--|
| 与用户痛点关联度 | 4 | 每个功能 / 动作可追溯到对应痛点 = 4；部分关联 = 2；无关联 = 0 |
| 与问题现状关联度 | 3 | 方案针对性解决问题现状 = 3；部分对应 = 1；无对应 = 0 |
| 方案完整性 | 3 | 覆盖核心场景无明显遗漏 = 3；有较多空白 = 1；残缺 = 0 |

> **无用功能识别（不减分，强制标记）：** 扫描方案中每个功能动作，若无法关联到任何用户痛点或问题现状，自动列入「无用功能清单」作为 🔴 阻塞项，要求 PM 说明理由或删除后再推进。

---

### B-6 目标 + 决策机制（15 分）⚠️ 关键项

| 子维度 | 满分 | 评分标准 |
|:--|:--|:--|
| 目标可量化 | 4 | 有具体数字 + 时间节点（如「上线 30 天内提升转化率 5%」）= 4；方向性目标 = 2；无目标 = 0 |
| 与解决方案关联度 | 4 | 每个目标可追溯到具体方案动作 = 4；部分关联 = 2；无关联 = 0 |
| 目标合理性 | 2 | 目标在资源范围内可达 = 2；目标明显不切实际 = 0 |
| 阶段化 Go/No-Go 机制 | 5 | 关键节点（如 T+30 / T+60 天）有具体数字阈值 + 触发条件 + 处置动作（继续 / 优化 / 中止）= 5；有目标无阶段分解 = 2；**无任何止损机制 = 0（阻塞项）** |

> **「成功就继续」不等于有决策标准。** 每项量化目标都需要配套说明：若未达标，是停止扩大投入？启动复盘？还是调整方案继续？没有止损机制的 MRD，决策者无法在项目执行途中做出有依据的判断。

---

### B-7 经济模型（条件性基础项，10 分）⚠️ 关键项

| 子维度 | 满分 | 评分标准 |
|:--|:--|:--|
| 成本估算完整性 | 4 | 有奖励成本 / 推广费用 / 研发工时的量化估算 = 4；有大致方向无数字 = 2；无 = 0 |
| 收益路径说明 | 4 | 说明成本如何转化为 GMV / 手续费 / 用户 LTV（如「奖励 X USDT → 预期拉新 Y 人 → 首 90 日人均 GMV Z → 手续费收入 W」）= 4；定性说「有商业价值」= 1；无 = 0 |
| 成本效益合理性 | 2 | LTV : CAC ≥ 4:1（行业合格线）或给出替代合理性论据 = 2；仅有数字无对比判断 = 1；无 = 0 |

> **适用条件（满足任一即为必填项）：**
> - 含任务奖励 / 活动奖励 / 合约体验金等直接成本
> - 涉及推广费用 / KOL 预算
> - 需要跨部门协调超过 20 人/日研发资源
>
> 不涉及以上情形的纯文案 / 纯规则类需求，本项视为满分。

## 加分项评分细则（3 项 × 10 分 = 30 分）

### P-1 市场机会（0–10 分）

| 子维度 | 满分 | 评分标准 |
|:--|:--|:--|
| 市场容量分析 | 5 | 有 TAM / SAM 数据 = 5；有定性判断 = 2；无 = 0 |
| 时机合理性 | 5 | 说明为什么现在做 = 5；仅说「市场需要」= 2；无 = 0 |

### P-2 风险预案（0–10 分）

| 子维度 | 满分 | 评分标准 |
|:--|:--|:--|
| 风险识别完整 | 5 | 覆盖技术 / 产品 / 业务 ≥ 3 类风险 = 5；仅 1 类 = 2；无 = 0 |
| 应对措施可行 | 5 | 每个风险有具体应对 = 5；有措施但模糊 = 2；无 = 0 |

### P-3 降级方案（0–10 分）

| 子维度 | 满分 | 评分标准 |
|:--|:--|:--|
| 降级场景定义 | 5 | 明确说明哪些条件下降级 = 5；模糊定义 = 2；无 = 0 |
| 降级方案完整 | 5 | 降级后用户体验有保障 = 5；降级方案残缺 = 2；无 = 0 |

## 问题输出规则

| 级别 | 标记 | 触发条件 | 处理方式 |
|:--|:--|:--|:--|
| 阻塞项 | 🔴 | 基础项任一得分 ≤ 2 分；B-6 无 Go/No-Go 机制；发现无用功能未说明；目标与方案无关联 | 必须处理完再推进，评审前重新提交评分 |
| 建议项 | 🟡 | 基础项某维度得分偏低但不影响核心逻辑 | 建议改进，可讨论是否本轮修复 |
| 优化项 | 🟢 | 加分项缺失或不完整 | 可选，不影响推进 |

## 综合评级

| 分数区间 | 评级 | 结论 |
|:--|:--|:--|
| 88–100 | ⭐ 优秀 | 可直接进入评审会 |
| 72–87 | 良好 | 补充建议项后评审 |
| 55–71 | 基础达标 | 修复阻塞项后重新提交 |
| 38–54 | 不合格 | 需重写关键章节 |
| < 38 | ❌ 重做 | 整体逻辑不成立 |

## 附：开发难度评估模块（独立输出，不计入总分）

基于方案内容独立输出，帮助研发团队快速掌握工作量全貌，并识别是否存在更小的 MVP 路径。

| 输出项 | 说明 |
|:--|:--|
| 功能拆解 | 将解决方案拆解为独立功能模块，逐项列出 |
| 开发工时预估 | 每个模块预估人/日工作量，汇总总工时区间 |
| MVP 建议 | 识别核心路径（P0 功能），给出最小可行方案，说明削减依据 |
