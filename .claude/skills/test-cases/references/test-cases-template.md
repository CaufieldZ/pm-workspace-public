# 测试用例集模板

> 本文件定义测试用例集的章节结构和填充规范。
> 生成测试用例时严格按此结构输出，不要增删章节。

---

## 文档结构

```markdown
# {产品名称} · 测试用例集 v{版本号}

> **权威版本**：本文档基于 behavior-spec `bspec-{项目简称}-v{N}.md` 和 PRD `prd-{项目简称}-v{N}.docx` 生成。如有冲突，以 PRD 为准。
> **面向读者**：测试 AI / QA 工程师
> **覆盖场景**：{列出 scene 编号}
> **用例统计**：正常 × N · 边界 × N · 异常 × N · 合规 × N · 待确认 × N

## 4. 边界条件用例

### 4.1 数值边界

| 编号 | 场景描述 | 前置条件 | 操作步骤 | 预期结果 | 裁剪依据 |

| TC-{编号}-B02 | {差一点未达标} | {数据设为边界值-1} | ... | {系统拒绝/提示未达标} | 单变量必覆盖 |

> 每个含数值阈值的业务规则都必须展开为 min / 边界 / max 三点。
> 阈值来源：behavior-spec 3.x.3 业务规则。

### 4.2 关键组合

| 编号 | 场景描述 | 前置条件 | 操作步骤 | 预期结果 | 裁剪依据 |
|------|----------|----------|----------|----------|----------|
| TC-{编号}-B10 | {多维度交叉场景} | {多个维度设为特定值} | ... | ... | {为什么选这个组合} |

> 多维度交叉不全排列，只取关键组合。
> 裁剪依据列必须填写：业务风险 / 用户频率 / 历史缺陷率。
```

## 填充规范

### 编号规则

- 格式：`TC-{Scene 编号}-{类型缩写}{序号}`
- 类型缩写：`N`（正常）/ `B`（边界）/ `E`（异常）/ `C`（合规）
- 示例：`TC-A-N01`、`TC-B1-B02`、`TC-C-E01`、`TC-M1-C01`
- 序号从 01 开始，同一 Scene + 同一类型内连续编号
- 边界条件的关键组合从 B10 开始编号（与单变量边界 B01-B09 区分）

### 前置条件

- 必须可独立准备：不依赖其他用例的执行结果
- 必须具体：不说「数据已准备」，说「已创建 1 个进行中的活动，奖励池余额 100 USDT，用户未参与」
- 引用术语表中的术语，不自创

### 操作步骤

- 用编号步骤：1. 2. 3. ...
- 只描述用户行为和系统交互，不描述 UI 细节（不写「点击右上角蓝色按钮」）
- 步骤粒度：一个步骤 = 一个用户操作 + 系统响应

### 预期结果

- **必须可判定**：测试 AI 读完后能直接判断 pass/fail
- **禁止模糊词**：「正常显示」「操作成功」「页面正常」「正常工作」
- **正确写法**：描述具体的可见结果和系统行为
  - ✅ 「显示绿色 Toast「报名成功」，卡片倒计时开始，按钮变为灰色「已报名」」
  - ✅ 「提示「余额不足」，金额输入框标红，提交按钮保持禁用」
  - ✅ 「KYC 等级校验不通过，弹出 KYC 升级引导弹窗，不进入下一步」

### 裁剪依据（仅边界条件关键组合需要）

- 必须注明为什么选这个交叉组合
- 依据优先级：业务风险 > 用户频率 > 历史缺陷率
- 示例：「用户频率高 + 历史上该组合出过 P1 缺陷」

### 待业务确认

- PRD 中标注「待定」「TBD」「需确认」的 → 必须列入
- behavior-spec 中缺失但 PRD 有提及的 → 列入
- 业务规则存在两种合理解读的 → 列入
- 不生成正式用例，只列测试关注点

## 切分多份文档时的规范

当按模块/场景组拆分为多份独立文档时，每份必须：

1. **独立的术语表**：只包含本模块用到的术语
2. **独立的环境前提**：本模块所需的数据和账号
3. **独立的编号段**：TC 编号在文档内连续
4. **各自通过自检**：每份文档独立执行自检清单

文件命名：`tc-{项目简称}-{模块名}-v{N}.md`
