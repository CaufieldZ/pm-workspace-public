# 测试用例集生成器 Skill（Test Cases）

## 作用

全局 PRD 完成并自检通过后，从 PRD 需求说明 + 场景清单提取测试维度，按单变量边界展开 + 成对覆盖（Pairwise）交叉组合，生成一份**测试用例集**。面向测试 AI 或测试人员直接执行。

核心价值：
- **四类全覆盖**：正常路径 / 边界条件 / 异常场景 / 合规校验，不遗漏
- **可验证的预期结果**：每条用例的预期结果都是可判定的，不用人工解读
- **Pairwise 建模**：P0 场景用成对覆盖算法裁剪组合，替代手工笛卡尔积裁剪
- **待确认项隔离**：PRD 未拍板的内容不混入正式用例，单独列出

## 输入源关系

| 输入 | 必须/可选 | 提取内容 |
|------|----------|---------|
| scene-list.md | 必须 | 场景编号、优先级、View 划分 |
| PRD §3.3 | 必须 | 需求说明、边界条件、异常处理、合规规则 |
| behavior-spec | 可选 | 补充异常场景 + 状态流转细节（有则用例质量更高） |

## 硬规则

### 必须包含

1. 正常路径：核心业务操作的完整流程
2. 边界条件：所有数值边界（恰好达标 vs 差一点未达标）
3. 异常场景：网络超时、权限不足、并发冲突等
4. 合规校验：受限地区、KYC 等级、风控规则
5. 待业务确认：PRD 未拍板的内容，单独成段

### 禁止写入

- 实现层语言（同 behavior-spec 的禁止清单）
- 点击坐标 / 元素 ID / 接口参数（属 UI 自动化范畴，不进业务用例集）
- XPath / CSS selector
- 业务策略和「为什么做」（属于全局 PRD）
- 排期和里程碑

### 预期结果写法

| 禁止 | 正确 |
|------|------|
| 正常显示 | 显示「申购成功」提示，余额减少对应金额 |
| 页面加载完成 | 页面展示活动列表，至少包含 1 张卡片 |
| 提示错误 | 顶部弹出红色 Toast「网络异常，请重试」，按钮恢复可点击状态 |
| 校验通过 | KYC 等级满足要求，用户进入下一步操作页面 |

原则：预期结果必须**可判定**——测试 AI 读完后能直接判断 pass/fail，不需要人工解读。

## 输出格式

**Markdown**（`.md`），存 `projects/{项目名}/deliverables/tc-{项目简称}-v{N}.md`

命名前缀 `tc-`，版本规则同其他产出物。

## 覆盖密度要求

| 场景优先级 | 最少用例数 | 覆盖要求 |
|-----------|----------|---------|
| P0 | ≥ 3 条 | 正常 + 至少 1 条边界 + 至少 1 条异常 |
| P1 | ≥ 2 条 | 正常 + 至少 1 条异常 |
| P2 | ≥ 1 条 | 正常路径即可 |

## 链路位置

复杂 / 超复杂链路中，**可选**步骤，PRD 自检通过后触发：

```
5. PRD
   ├── 6a. behavior-spec（可选）
   ├── 6b. page-structure（可选）
   └── 6c. 测试用例集（可选，可与 6a/6b 并行）
7. 拉通自检
```

前置条件：全局 PRD 自检通过。有 behavior-spec 时优先读取以补充异常覆盖。

## 执行步骤

### Step 1：读取输入

**必读**（并行读取）：

```
view projects/{项目名}/context.md
view projects/{项目名}/scene-list.md
```

**PRD 读取**（主输入源）：
- PRD §3.3 需求说明：边界条件、异常处理、业务规则
- PRD 合规/风控章节：地区限制、KYC 要求、风控策略
- docx 格式：`python3 -c "from docx import Document; ..."` 提取对应章节
- md 格式：grep 定位后 Read 局部

**behavior-spec 补充读取**（如已生成）：
- `projects/{项目名}/deliverables/bspec-{项目简称}-v{N}.md`
- 补充异常场景 + 状态流转细节

### Step 1.5：多维参数建模（P0 场景必做，≤ 2 个输入维度可跳过）

对每个 P0 场景，在写用例之前先做等价类划分：

1. **识别输入维度**：找出该场景的关键变量（地区 / KYC 等级 / 产品类型 / 余额状态 / 用户角色 / 时间窗口等）
2. **等价类划分**：每个维度用业务语义分组，不写具体数值（如「余额状态: 零值, 低于最低限额, 充足」而非「0, 50, 200」）
3. **列出约束规则**：哪些组合业务上不允许（如「受限地区 + 任何产品类型 = 不允许」）
4. **生成参数组合矩阵**（≥ 3 个维度时推荐）：调用 `scripts/pairwise_gen.py` 生成成对覆盖组合，把矩阵每行扩展为一条完整用例

```bash
# 示例：在脚本顶部定义参数和约束后运行
python3 .claude/skills/test-cases/scripts/pairwise_gen.py demo
```

参数建模完成后，再进入 Step 2 按类型写用例。参数矩阵中的每行 = 至少一条边界/异常用例。

### Step 2：提取测试维度

从 behavior-spec 提取以下测试维度：

| 维度 | 来源 | 提取方法 |
|------|------|----------|
| 操作 | 3.x.1 用户行为 | 逐行提取操作名 |
| 状态 | 2.2 全局状态流转 | 提取所有状态名 |
| 角色/权限 | 2.1 角色与权限 | 提取角色和可见范围 |
| 数值边界 | 3.x.3 业务规则 | 提取含数值的规则（阈值、限制、公式） |
| 异常类型 | 3.x.4 异常场景 | 提取异常场景列表 |
| 合规规则 | PRD 业务规则章节 | 提取 KYC/地区/风控规则 |

### Step 3：展开用例

**展开策略**：

1. **正常路径**：每个操作 × 默认态 = 1 条用例
2. **边界条件**：每个数值规则展开为 min/边界/max 三点
3. **异常场景**：每个操作 × 每个相关异常 = 1 条用例
4. **合规校验**：每个合规规则 × 满足/不满足 = 至少 2 条用例

**交叉组合裁剪规则**：
- 单变量边界**必须覆盖**（不可省略）
- 多维度交叉**只取关键组合**（最可能出问题的 2-3 个），不全排列
- 裁剪依据：业务风险 > 用户频率 > 历史缺陷率
- 每个交叉组合用例必须在「裁剪依据」列注明原因

**待业务确认项**：
- PRD 中明确标注「待定」「TBD」「需确认」的规则 → 不生成正式用例
- behavior-spec 中缺失但 PRD 有提及的规则 → 列入「待业务确认」段
- 业务规则中存在歧义（两种解读都合理）→ 列入「待业务确认」段

### Step 4：生成测试用例

严格按 `references/test-cases-template.md` 的章节结构填充。

用例编号规则：
- 格式：`TC-{Scene 编号}-{类型缩写}-{序号}`
- 类型缩写：`N`（正常）/ `B`（边界）/ `E`（异常）/ `C`（合规）
- 示例：`TC-A-N01`、`TC-B1-B02`、`TC-C-E01`、`TC-M1-C01`

### Step 5：自检

按下方自检清单逐项验证，全部通过后才可交付。

### Step 6：保存 & 通知

保存到 `projects/{项目名}/deliverables/tc-{项目简称}-v{N}.md`

通知用户：
- 文件路径
- 覆盖的场景编号列表
- 四类用例数量统计
- 待业务确认项数量

## 自检清单

- [ ] **覆盖密度达标**：P0 场景 ≥ 3 条，P1 场景 ≥ 2 条，P2 场景 ≥ 1 条
- [ ] **四类用例全覆盖**：正常路径 / 边界条件 / 异常场景 / 合规校验，每类至少 1 条
- [ ] **前置条件明确**：每条用例的前置条件可独立准备，不依赖其他用例的执行结果
- [ ] **预期结果可验证**：预期结果不含「正常显示」「操作成功」等模糊表述，全部可判定 pass/fail
- [ ] **无 UI 自动化细节**：不出现坐标、元素 ID、XPath、CSS selector
- [ ] **无实现层语言**：grep 检查无接口路径、数据库字段名、精度类型
- [ ] **与 behavior-spec 一致**：操作名、状态名、术语与 behavior-spec 完全相同
- [ ] **待确认项已隔离**：PRD 未拍板内容单独列在末尾，不混入正式用例
- [ ] **编号与 scene-list 一致**：TC 编号中的 Scene 编号可逐个比对
- [ ] **数值边界三点覆盖**：每个含阈值的规则都有 min/边界/max 对应用例

**强制验证脚本**：

```bash
FILE="projects/{项目名}/deliverables/tc-{项目简称}-v{N}.md"
echo "=== 禁止项检查 ==="
grep -n 'xpath\|css.*selector\|getElement\|click.*[0-9]\{2,\}\|/api/\|Float\|INT\|VARCHAR\|Redis\|MySQL' "$FILE" && echo "❌ 发现禁止项" || echo "✅ 无禁止项"

echo "=== 四类用例覆盖检查 ==="
for tag in '-N' '-B' '-E' '-C'; do
  COUNT=$(grep -c "TC-.*${tag}" "$FILE" 2>/dev/null || echo 0)
  echo "  类型 ${tag}: ${COUNT} 条"
done

echo "=== 预期结果模糊词检查 ==="
grep -n '正常显示\|操作成功\|页面正常\|正常工作\|正常加载' "$FILE" && echo "❌ 发现模糊预期结果" || echo "✅ 预期结果可验证"

echo "=== 待确认项隔离检查 ==="
grep -q '待业务确认\|待确认' "$FILE" && echo "✅ 待确认项已单独列出" || echo "⚠️ 无待确认项（如 PRD 无待定内容则正常）"

echo "=== 前置条件检查 ==="
grep -c '| 前置条件' "$FILE" | xargs -I{} echo "  含前置条件的行数: {}"
```
