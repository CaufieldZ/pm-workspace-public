# Metrics Framework · 商业增长指标方法论

PRD 1.2 节四件套（北极星 / 配套 / Guardrail / 非目标）的写作方法论。回答「PM 怎么想清楚要追求什么、不能毁掉什么、不追求什么」，不是「怎么写代码」。

四件套的代码生成由 [`sections.py`](../scripts/sections.py) `north_star_section` helper 负责，本文档只管方法论；helper 不做语义校验（不强制北极星格式、不强制 Guardrail 至少几项），靠 PM 在写之前读本文档拿对方法。

## 何时读本文档

- **PRD 写作前必读**：写 1.2 北极星章节之前先过一遍方法 + 反例集（prd skill Step 1 已强制）
- **context.md 第 2 章「项目目标」起草时建议读**：让目标讨论从立项就用同一套方法论
- **PM-GATE 必答三问回答前建议读**：「核心指标是什么 / 怎么判断成功」两问按本文档校准

不适用：埋点事件命名（归 PRD 第 8 章 / data-report skill）、A/B 实验设计（独立 skill）、归因模型（数据团队主导）。本文档**仅**覆盖 1.2 节四件套的方法论。

## 一、为什么强制四件套

PRD 1.2 节常见反例：写一句「提升用户体验 / 增加用户粘性 / 提升 DAU」就过。一旦上线后没法判断成败 —— 三个月后 DAU 涨了 5%，是项目效果还是大盘自然涨？涨了但 Crash 也涨了 30%，算不算成功？运营加了 push 把 DAU 拉起来了，跟 PRD 改的功能有关吗？

四件套强制 PM 在写 PRD 前同时想清楚四个问题：

| 段 | 回答 | 没写的代价 |
|---|---|---|
| **北极星** | 单一最关键指标 + 现状值 → 目标值 + 时间窗 | 上线后没法量化判断 |
| **配套** | 2-3 个支撑北极星的中间指标 | 北极星没动时不知道哪一环出问题 |
| **Guardrail** | 本期不能恶化的指标 | 砍掉首屏其他模块换 DAU 算赢吗？ |
| **非目标** | 本期明确不追求的方向 | 评审会上反复被问「为啥不顺手优化 X」 |

**北极星 + Guardrail 是一对**。北极星只看正向、看「赚什么」；Guardrail 强制 PM 对齐「为了赚 X 不能赔 Y」。Crash / 客诉 / 留存是默认三件套，按业务再加。

**非目标 ≠ 没考虑**。非目标是显式声明「想过了，本期不做，原因是 X」。空着或写「无」三种可能：① 真没想 ② 想了但不敢承诺 ③ 想了但藏着 —— 三种都坏。

## 二、北极星指标 · 选型方法

### 单一性铁律

每个 PRD **只有一个**北极星。两个就不是「最关键」。如果真的有两个候选纠结不下，问自己：上线 30 天后只能挑一个数字汇报，挑哪个？

常见违反：

```
❌ 北极星：DAU + GMV + NPS 都要涨
✅ 北极星：活动中心 DAU 12K → 25K（3 个月内）
   配套：GMV 关联指标走配套段（不是北极星）
   配套：NPS 走配套段或不写（NPS 噪音大、慢，不适合做北极星）
```

### 三段式格式（强制）

`[场景限定] [指标名] [现状值] → [目标值]（[时间窗]）`

每一段缺了都是漏洞：

| 漏的字段 | 反例 | 后果 |
|---|---|---|
| 场景限定 | 「DAU 25K」 | 全站 DAU 还是活动中心 DAU？ |
| 现状值 | 「DAU 涨到 25K」 | 评审看不出难度，从 1K 涨到 25K 跟从 24K 涨到 25K 完全不是一个项目 |
| 目标值 | 「DAU 显著增长」 | 上线后没法判断 |
| 时间窗 | 「DAU 12K → 25K」 | 上线第二天就拍数还是稳态？ |

### 选型四个判据

选北极星指标时按下列优先级判断（前者满足才考虑后者）：

1. **可归因**：北极星动了，能不能归因到本 PRD 的改动？
   - 反例：选「整站 DAU」做活动中心 PRD 的北极星 —— 整站 DAU 受运营 push、新币上线、合约行情主导，活动中心改版的归因占比可能 <5%
   - 正例：选「活动中心 DAU」—— 改版直接影响这个数

2. **可测量**：现有埋点 / BI 能取到吗？取数延迟 ≤ 1 周吗？
   - 反例：选「活动忠诚用户占比」但忠诚定义没敲定 + 取数要离线跑一周
   - 正例：选「活动报名率」—— 报名按钮点击 / 详情页 PV 已有埋点

3. **可控制**：本 PRD 改动能直接撬动这个数吗？
   - 反例：选「活动转化率」但本期只改首页入口 —— 转化率受详情页 + 报名流程主导，入口对它的影响 <10%
   - 正例：选「活动入口 CTR」—— 改入口直接打这个

4. **业务相关**：这个数涨了，业务方真的认吗？
   - 反例：选「活动详情页 PV」—— PV 涨可能是用户进来又退出，业务不认
   - 正例：选「活动报名率」—— 报名了业务方认为是有效用户

### 长 vs 短期

北极星看 PRD 类型选周期：

| PRD 类型 | 时间窗 | 例子 |
|---|---|---|
| 触达 / 入口型 | 上线后 7-14 天稳态 | 活动中心入口 CTR |
| 转化 / 流程型 | 上线后 14-30 天稳态 | 活动报名率 |
| 留存 / 习惯型 | 上线后 30-90 天稳态 | 活动 D7 回访率 |
| 收入 / 商业化 | 上线后 30-60 天稳态（剔除上线尝鲜期） | 活动 GMV |

「7-14 天」「30-90 天」要写到时间窗里，不能只写「3 个月内」。

## 三、配套指标 · 中间指标推导

### 数量：2-3 个

少于 2 个 → 北极星没动时无法定位哪一环失败。多于 3 个 → 评审注意力被稀释，等于没有。

### 推导方式：北极星拆漏斗

把北极星拆成「上游 → 中游 → 下游」三段，挑 2-3 个能代表关键节点的中间指标。

```
北极星：活动中心 DAU 12K → 25K（3 个月内）
拆漏斗：
  上游 = 入口曝光  → 配套 1：首页活动 Tab CTR ≥ 8%（现状 5.2%）
  中游 = 进入活动  → 配套 2：活动详情页转化率 ≥ 15%（现状 9%）
  下游 = 留住用户  → 配套 3：我的活动 D7 回访率 ≥ 40%
```

漏斗任一节点出问题都能从配套指标看出来：CTR 没涨 → 入口设计有问题；CTR 涨但转化率没涨 → 详情页有问题；转化率涨但 D7 没涨 → 活动质量有问题。

### 不要选这种配套

| 反例 | 为什么不行 |
|---|---|
| 「用户体验 NPS ≥ 4.5」 | NPS 噪音大、慢、跟北极星归因弱 |
| 「页面加载性能 ≤ 1.5s」 | 这是 Guardrail 不是配套（性能不是「支撑北极星」是「不能恶化」） |
| 「活动数 ≥ 30 个 / 月」 | 这是运营投入指标不是产品指标，不归 PRD 管 |
| 「用户满意度调研得分 ≥ 4 分」 | 调研有偏 + 周期长，不能跟北极星打平 |

## 四、Guardrail · 反向指标

### 默认三件套

每个 PRD 按下列默认值起草（按业务再加）：

| Guardrail | 阈值起草思路 | 数据源 |
|---|---|---|
| **本端 Crash 率** | 现状值不退化（如现状 0.18‰ → 红线 0.3‰） | APM / 大盘 Crash |
| **客诉率** | 现状值 +10% 以内（如 4.2/万 DAU → 红线 4.6/万 DAU） | 客服系统按 PRD 模块筛 |
| **整站 D1 留存** | 不下降（如现状 78% → 红线 78%） | BI |

### 业务定制

按 PRD 类型加：

| 类型 | 加的 Guardrail |
|---|---|
| 改首屏 / 改入口 | 首屏其他模块 CTR 不能下降（避免抢流量） |
| 改交易流程 | 交易成功率不能下降、错单率不能上升 |
| 改资金 / 出入金 | 资金对账误差 = 0、超时订单率不上升 |
| 改通知 / push | 取关率不上升、用户屏蔽率不上升 |
| 改算法 / 推荐 | 内容多样性指数不下降、长尾内容曝光占比不下降 |
| 改合规相关 | 合规审查通过率 = 100%、敏感信息泄漏 = 0 |

### 阈值怎么定

不要拍脑袋写「Crash < 0.5‰」。三步：

1. **取现状值**：跑 BI / APM 拿过去 30 天的均值 + P95
2. **画红线**：均值 + (P95 - 均值) × 1.5（给业务波动留余量）
3. **触发响应**：超阈值后做什么写到 7.x Risks（监控告警 + 立即回滚 / 灰度回退一档）

### 写成「不下降」还是「不超过 X」

- 现状值波动小、可量化 → 写绝对值（「Crash < 0.3‰」）
- 现状值波动大、按相对变化看 → 写相对（「客诉率 ≤ 现状 +10%」）
- 强烈推荐写绝对值，别让评审跟你查现状值再算

## 五、非目标 · 边界声明

### 三类高频非目标

每个 PRD 看下列清单，不在本期范围的显式声明：

| 类型 | 例子 |
|---|---|
| 性能 / 加载优化 | 「不优化首屏加载性能（现状满足 SLA，性能优化排在 V3）」 |
| 端 / 平台覆盖 | 「不做 Web 端（本期 App-only，Web 排在 P1）」 |
| 流程 / 合规改造 | 「不改 KYC 流程（法规审查窗口未开）」 |
| 数据 / 分析延伸 | 「不做归因看板（归独立 BI 项目）」 |
| 字段 / 业务系统侧 | 「不动业务系统侧字段定义（同步契约只读）」 |
| 跨业务线扩展 | 「不重构合约活动中心（独立项目 htx-futures-zone）」 |

### 每条非目标必带「原因」

格式：`[范围]，[原因（资源 / 优先级 / 技术风险 / 法规 / 业务边界）]`

```
❌ 不优化首屏性能
✅ 不优化首屏性能 — 现状 1.8s 满足 SLA，性能优化排在 V3（H2）

❌ 不做 Web 端
✅ 不做 Web 端 — 本期资源仅覆盖 App，Web 排在 P1，预计 V2.1 启动
```

原因写清楚，后续被问起来不用临时编。

### 真的没有非目标怎么办

罕见。如果真没有，显式写「**本期无 Non-goals（本期所有需求都做）**」并在评审主动说明为什么 —— 这一行会引出评审挑战，让边界讨论暴露在台面上而不是上线后才被问。

## 六、反例 vs 正例 · 完整对照

### 反例 · 模糊不可验证

```
1.2 核心目标
- 提升活动中心使用率
- 优化用户体验
- 增加用户粘性
- KPI：DAU 增长、留存提升
```

问题清单：

1. ❌ 没现状值 → 评审看不出难度
2. ❌ 没目标值 → 上线后没法判断成败
3. ❌ 没时间窗 → 上线第二天还是稳态？
4. ❌ 没 Guardrail → 砍掉首屏其他模块换 DAU 算赢吗？
5. ❌ 没非目标 → 评审会上反复被问边界
6. ❌ 「优化用户体验」「增加用户粘性」是套话，跟没写一样

### 正例 · 四件套齐全（活动中心 v5.3 实际样本）

```
**北极星指标**
- 活动配置错误率从 v5.1 的 18%（多端不一致 / 翻译漏配 / 跳转错位）
  降到 v5.3 上线 30 天内 ≤ 5%

**配套指标**（2-3 个）
- 运营配置一个活动的平均耗时从 25 分钟降到 ≤ 8 分钟
- 用户端活动卡片点击率（CTR）相对 v5.1 提升 ≥ 15%
- 「我的活动」页 7 日回访率 ≥ 30%（v5.0 独立页化后基线）

**反向指标（Guardrail · 不能恶化）**
- 同步延迟 P95 ≤ 5min，超过则降级走运营手动建活动
- 前端首屏渲染性能不能恶化（不增加同步阻塞调用）
- MGT 配置页 Crash < 0.3‰

**非目标**
- 本期不动业务系统侧字段定义（同步契约只读）
- 本期不做活动数据分析报表（归独立 BI 项目）
- 本期不重构合约活动中心（独立项目 htx-futures-zone）
```

为什么这个样本算合格：

1. ✅ 北极星单一 + 含场景限定（活动配置错误率）+ 现状值（18%）+ 目标值（≤ 5%）+ 时间窗（30 天内）
2. ✅ 配套 3 个，分别覆盖运营效率（耗时）/ 用户触达（CTR）/ 用户留存（D7 回访），漏斗节点齐全
3. ✅ Guardrail 3 项，覆盖技术约束（同步延迟）/ 副作用约束（首屏性能）/ 默认三件套之一（Crash）
4. ✅ 非目标 3 项，每项带原因（业务系统契约 / 项目分工 / 跨项目边界）

## 七、自检清单

写完 1.2 节自查 8 项，全过才算合格：

| 项 | 通过条件 |
|---|---|
| ① 北极星单一 | 只有 1 个，不是 2-3 个并列 |
| ② 北极星三段式 | 含场景限定 + 现状值 + 目标值 + 时间窗 |
| ③ 北极星可归因 | 本 PRD 改动能直接撬动该指标 |
| ④ 配套 2-3 个 | 不少于 2 不多于 3 |
| ⑤ 配套覆盖漏斗 | 上游 / 中游 / 下游分别有代表性指标 |
| ⑥ Guardrail ≥ 3 项 | 含默认三件套或同等级业务 Guardrail |
| ⑦ 非目标 ≥ 1 项 | 真无则显式写「本期无 Non-goals」 |
| ⑧ 非目标带原因 | 每条都说明为什么不做 |

⑥ Guardrail ≥ 3 项是软标准（helper 不强制），但只写 1 项基本说明 PM 没认真想边界。

## 八、跟 north_star_section helper 的接力

`sections.north_star_section` 只负责把 4 个 list 渲染成 docx 段落，不做语义校验。本文档定义的方法论由 PM 在写代码前内化：

```python
from sections import north_star_section

# 写之前先按本文档自检 8 项
north_star_section(
    doc,
    north_star="...",       # 三段式：场景限定 + 现状 → 目标 + 时间窗
    supporting=[...],       # 2-3 个，覆盖漏斗
    guardrail=[...],        # ≥ 3 项，含默认三件套
    non_objectives=[...],   # ≥ 1 项 + 每条带原因
)
```

helper 内部只检：

- north_star 非空（PR4 落地）
- guardrail 非空（save_prd 体检扫 1.2 段是否含 Guardrail / 反向指标 关键词）
- non_objectives 空时自动写占位「本期无 Non-goals」

**why helper 不做更严的校验**：方法论靠 PM 内化，工具校验越严越容易触发误报（北极星格式自由度高，正则只能扫到表面合规）。本文档是一切之根，helper 是渲染壳。
