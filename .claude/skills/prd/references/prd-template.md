# PRD 模板 · 分档参考

PRD 不是单一模板。按需求复杂度匹配体量，简单需求一页搞定，复杂跨系统才上完整产品思维要素。

## 何时用哪档（强制判档）

| 档位 | 触发 | 章节 | 体量 | 推 wiki |
|------|------|------|------|---------|
| **极简 One-pager** | ≤ 3 场景 / 单端 / 纯文案 / 配置项 | 4 节 Markdown | ~1 页 | `scripts/md_to_confluence.py` |
| **标准** | 4-10 场景 / 2-3 端 | 当前九章 + 1.5 Non-goals + 6.x Open Questions（必填） | 当前体量 | `push_to_confluence_base.py` |
| **完整** | 10+ 场景 / 多系统 / 跨团队 / 资金流转 | 标准 + 1.6 方案选型 + 7.x Risks & Mitigations | 略大 | 同标准 |

PM 在 PM-GATE 后明确档位，或 SKILL Step 2 自动建议（按 scene-list 行数 + view 数 + 是否跨系统判）。**没拿准时往上一档**（保守优于事后补章节）。

## 极简档 · One-pager Markdown

### 适用

- ≤ 3 场景，单端
- 纯文案 / 策略 / 配置项调整
- 无原型 / 无截图依赖

### 模板

```markdown
# [产品 / 功能名] PRD（极简档）

| 属性 | 内容 |
|------|------|
| 撰写日期 | YYYY-MM-DD |
| 产品线 | ... |
| 状态 | 评审中 / 已上线 |

## 1. 背景与目标

[2-3 句现状痛点 + 这次为什么做]

**北极星指标**：[现状 → 目标 + 时间窗]

**反向指标（Guardrail）**：[不能恶化的，如 Crash / 客诉 / 留存]

**非目标**：[本期明确不追求的]

## 2. 场景与方案

### 2.1 [场景 A]
- 触发方式：...
- 操作步骤：...
- 异常态：...

### 2.2 [场景 B]
（同上）

## 3. 业务规则

| 规则 | 详情 |
|------|------|
| ... | ... |

## 4. 度量与发布

- 上线节奏：[灰度 / 全量]
- 核心监控：[北极星 + Guardrail 的具体监控点]
- 回滚预案：[条件 + 操作]
```

### 推送

```bash
python3 scripts/md_to_confluence.py <md> --parent-id <id>
```

`pre-wiki-push-gate` hook 对 md 走 warn 模式（不强阻断），自检靠 `post-cjk-punct-check` hook + 人工 review 兜底。

## 标准档 · docx 九章

### 章节结构总览

```
封面 + 元信息表
├── 1. 项目背景与目标
│   ├── 1.1 背景
│   ├── 1.2 核心目标 · 北极星 / 配套 / 反向 / 非目标（必填）★ 升级
│   ├── 1.3 变更范围（迭代项目必填）
│   ├── 1.4 用户角色
│   ├── 1.5 Non-goals（必填，标准 + 完整档）★ 新增
│   └── 1.6 方案选型说明（完整档选填）★ 仅完整档
├── 2. 场景地图
│   ├── 2.1 View 1 · [名称]
│   ├── 2.2 View 2 · [名称]
│   └── 2.3 View 3 · [名称]
├── 3. [View 1] 详细需求
├── 4. [View 2] 详细需求
├── 5. [View 3] 详细需求
├── 6. 业务规则
│   ├── 6.1 [规则类型]
│   ├── 6.2 枚举值列表
│   ├── 6.X 组件复用 & 数据关系
│   ├── 6.X 页面层级
│   └── 6.X Open Questions（必填，无未决项写「无」）★ 新增
├── 7. 非功能性需求
│   └── 7.X Risks & Mitigations（完整档必填，标准选填）★ 新增
├── 8. 埋点与监控
├── 9. 里程碑与排期
│   └── 9.1 后续规划
└── — 文档结束 —
```

**编号兼容性**：1.4 用户角色编号不变，Non-goals / 方案选型作为 1.5 / 1.6 追加在末尾，老 PRD 升版只追加新章不破现有结构。

### 第 1 章 · 项目背景与目标

#### 1.1 背景

[2-3 句话描述现状痛点和做这个项目的原因]

#### 1.2 核心目标 · 北极星 / 配套 / 反向 / 非目标（必填）★ 升级

每个 PRD 强制写四类指标，逼 PM 同时想清楚「要追求什么」「不能毁掉什么」「不追求什么」。

> 完整方法论见 [`metrics-framework.md`](metrics-framework.md)：北极星三段式 + 选型四判据 / 配套漏斗推导 / Guardrail 默认三件套 + 业务定制 / 非目标三类高频项 + 自检 8 项 + 反例对照。**写代码前先读**。

```
**北极星指标**：[单一最关键指标 + 现状值 → 目标值 + 时间窗]
  例：活动中心 DAU 12K → 25K（3 个月内）

**配套指标**（2-3 个）：[支撑北极星的中间指标]
  例：首页 CTR、活动报名率、各漏斗节点留存

**反向指标（Guardrail）**：[本期不能恶化的指标]
  例：首页 Crash < 0.3‰；客诉率 ≤ 现状 +10%；整站留存不下降

**非目标**：[本期明确不追求的方向]
  例：不优化加载性能；不改 KYC 流程
```

**为什么必填反向指标**：北极星只看正向，不看代价。Guardrail 强制 PM 对齐「为了赚 X 不能赔 Y」。Crash / 客诉 / 留存是默认三件套，按业务再加。

代码生成：

```python
from sections import north_star_section

north_star_section(
    doc,
    north_star="活动中心 DAU 12K → 25K（3 个月内）",
    supporting=["首页 CTR ≥ 8%", "活动报名率 ≥ 15%"],
    guardrail=["首页 Crash < 0.3‰", "客诉率 ≤ 现状 +10%", "整站留存不下降"],
    non_objectives=["不优化加载性能", "不改 KYC 流程"],
)
```

#### 1.3 变更范围（迭代项目必填，新项目可省略）

描述本次 PRD 相对**当前线上版本**的差异，不是 PRD 版本之间的 diff。PM 内部多轮迭代痕迹不入此章。

首行声明基线：

```
基线：V[x.x]（YYYY-MM-DD 上线）
```

基线选取规则：第 N 期迭代以「第 N-1 期已发布线上版本」为基线，不以最初版本为基线，避免 delta 越拖越长。

变更项格式：

```
- [模块/组件] [变更动作]（从现状 xxx 改为 xxx）
- [模块/组件] [砍掉/新增]，[说明]
```

新项目首发：直接写「本次为新项目首发，无基线变更（后续迭代以 v{当前版本} 上线版本为基线）」。

#### 1.4 用户角色

```
| 角色     | 端      | 描述               | 核心操作                     |
|----------|---------|--------------------|-----------------------------|
| 普通用户 | Web/App | 浏览、使用功能      | 浏览列表、点击操作、查看结果 |
| 运营人员 | 后台Web | 管理配置            | 上下线、编辑、配置规则       |
| 业务系统 | 服务端  | 自动同步数据        | 推送数据到前端系统           |
```

#### 1.5 Non-goals（必填，标准 + 完整档）★ 新增

**写「不做什么」比写「做什么」更难**，强迫 PM 跟自己拉锯一遍边界。无 Non-goals 时显式写「**本期无 Non-goals**」，不能留空。

```
| 范围 | 原因 |
|------|------|
| [范围 1] | [资源 / 数据 / 优先级 / 技术风险 / 法规等] |
| [范围 2] | ... |
```

例：

```
| 范围 | 原因 |
|------|------|
| 不优化首屏加载性能 | 现状 1.8s 满足 SLA；性能优化排在 V3 |
| 不改 KYC 流程 | 法规审查窗口未开 |
| 不做 Web 端 | 本期资源仅覆盖 App，Web 排在 P1 |
```

代码生成：

```python
from sections import non_goals_section

non_goals_section(doc, items=[
    ("不优化首屏加载性能", "现状 1.8s 满足 SLA；性能优化排在 V3"),
    ("不改 KYC 流程", "法规审查窗口未开"),
])
# 空列表 → 自动生成「本期无 Non-goals」占位
non_goals_section(doc, items=[])
```

#### 1.6 方案选型说明（仅完整档，2 选 1+ 时必填）★ 新增

回答「为什么选 A 不选 B/C」。让评审看到推理链而非结论。多方案讨论流水归 `context.md` 第 7 章，本章只写**最终态决策矩阵**。

```
| 方案 | 描述 | 优势 | 劣势 | 选/弃 |
|------|------|------|------|-------|
| A 当前方案 | ... | ... | ... | ✓ |
| B 替代方案 1 | ... | ... | ... | ✗（理由） |
| C 替代方案 2 | ... | ... | ... | ✗（理由） |
```

单方案直出免本章。完整档触发条件（多系统 / 跨团队 / 资金）通常意味着方案选型重要。

### 第 2 章 · 场景地图

第 2 章是「主线 → 模块」的桥梁：2.1 一句话 + 主线图讲完整故事、2.2 拆到每端给研发、2.3 把跨端衔接焊起来。

完整叙事投影规则见 [`prd-storytelling.md`](prd-storytelling.md) 第 2 章。

#### 2.1 用户旅程主线

一段话讲清主线（背景 → 冲突 → 关键路径 → 收束），**跟 context.md 第 4 章「场景叙事顺序」逐字对齐**。涉及 ≥ 2 端项目时，必须嵌入一张主线流程图。

```
[一段话主线，参 context.md 第 4 章顶部「场景叙事顺序」]

[嵌入图：deliverables/flow-{项目}-journey-v{N}.html 截图，由 flowchart skill 产出]
```

**生成方式**（PR4 落地后）：

- 先用 flowchart skill 产出 `flow-{项目}-journey-v{N}.html`（独立可交互产物）
- 写 PRD 时调 `journey_main_section(doc, narrative, flowchart_html_path)`，自动 Playwright 截图嵌入

#### 2.2 端 / 角色拆分总览

一句话总结每端职责，再给场景地图索引表。这一节把 2.1 主线落到第 3/4/5 章模块视图。

```
**每端职责：**
- View 1 · [端名]：[一句话职责]
- View 2 · [端名]：[一句话职责]
- View 3 · [端名]：[一句话职责]

| 编号 | 场景                          | 模块   | P  |
|------|-------------------------------|--------|----|
| A    | [场景名称]                     | [模块] | P0 |
| B-1  | [场景名称]                     | [模块] | P0 |
| B-2  | [场景名称]                     | [模块] | P0 |
```

**编号规则**：

- 主场景：大写字母 A / B / C / D...
- 子场景：字母-数字 B-1 / B-2 / B-3
- 后台场景：M-1 / M-2 / M-3
- 跨 View 场景：功能缩写前缀 F-0 / F-1（Futures）、E-1（Entry）

#### 2.3 跨端时序

关键的跨端交互单独出图（PRD docx 嵌图走 mermaid，按场景选类型：跨端协作 + Yes/No 分支用 `flowchart LR + subgraph` 分组流程图；强调时序消息 / 请求-响应往返用 `sequenceDiagram`；多状态切换用 `stateDiagram-v2`）。**分模块写第 3/4/5 章时跨端逻辑会被切碎**，每端只看到半边，时序图把它们焊起来。QA 写测试用例时这一节是黄金素材。

涉及 ≥ 2 端项目时**强制至少 1 张跨端流程图**。严格 swimlane 泳道图 mermaid 不直接支持，目前默认走 `flowchart LR + subgraph`（不是严格泳道，是分组流程图），视觉上够用；真要严格泳道得回归 X6 swimlane（视觉差，慎用）。

```
**跨端关键交互：**
- [交互名 1]（[发起端] → [接收端]，例：观众举手 → 主播看到 → 主播同意）
- [交互名 2]（异常态：例：上麦中断时主播 / 观众如何感知）
- [交互名 3]（多方同步：例：运营踢人 → 主播端 + 观众端三方同步）

[嵌入图：deliverables/flow-{项目}-cross-end-{N}.html 截图，可多张]
```

**生成方式**（PR4 + 任务 1/2 mermaid 化后）：

- 项目脚本里定义 mermaid 源码常量（`MERMAID_M7_CROSS_END = """flowchart LR\\n  ..."""`），按场景选 `flowchart LR + subgraph` / `sequenceDiagram` / `stateDiagram-v2`
- 写 PRD 时调 `cross_end_sequence_section(doc, sequences=[(标题, mermaid_src, note), ...])` 一对一独立渲染嵌入
- 渲染由 PRD skill 内部 `mermaid_screenshots.render_mermaid` 完成（DPI 200 + sha1 缓存），flowchart skill 不服务 PRD docx 场景

### 第 3/4/5 章 · 各 View 详细需求

章标题：「X. [View 名称] 详细需求（View N）」

**章首必须加用户故事引言**（pm-workflow.md 强制）：紧跟 h1 调用 `chapter_story(doc, "...")`，一句话 ≤ 30 字，告诉读者这章在讲谁、做什么、为什么。例：

- `chapter_story(doc, "用户进入个人主页查看战绩、收益与动态，是个人空间的核心承载。")`
- `chapter_story(doc, "运营在 CMS 配置活动卡片与白名单，决定前台用户看到的内容。")`

紧接着可保留排版斜体说明：*左侧为原型截图占位，右侧为功能需求说明。*

第 1 章背景目标 / 第 6 章业务规则 / 第 7 章非功能 / 第 8 章埋点 / 第 9 章排期为技术骨架章，免除此要求。

#### 两种 Scene 展开模式

**模式 A · scene_table（2 列，默认）**：用于**有独立 UI / 可截图**的 Scene（APP / Web / 有页面改动的后台）。左列原型截图 + 右列 numbered list 说明。

**模式 B · body 级 H3 + numbered list（无截图降级）**：用于 **CMS / 纯后台 / 本次 UI 不改只改后端 / 规则说明**等无独立截图的 Scene。不要用 2 列表格给空截图留占位符，直接：

```
## 7.X Scene F-N · [场景名]
（可选）**入口**：[一句话描述]

### [小节标题 1]
1. **条目类型**：内容
2. **条目类型**：内容

### [小节标题 2]
1. **条目类型**：内容
```

**判断准则**：Scene 改动范围内没有 UI 可以截图 → 用模式 B；有 UI 改动（哪怕只改 1 个按钮）→ 用模式 A 配截图。本次迭代 UI 不改但要讲历史现状的场景（如 CMS「保留现状 + 后端改动」）也用模式 B。

**代码实现**：模式 B 用 `insert_scene_blocks(anchor, blocks, heading_level=3)`（来自 `update_prd_base`），视觉与 scene_table 右列对齐。

#### 模式 A · Scene 详细说明模板（两列表格）

每个 Scene 用一个两列表格：

```
| [截图占位]                              | [功能需求]                                  |
|-----------------------------------------|---------------------------------------------|
| 🖥 A · [场景名] ← 此处粘贴原型截图      | **[模块名1]（变更/新增）**                   |
|                                         | 1. **[条目类型]**：[具体说明]                |
|                                         | 2. **[条目类型]**：[具体说明]                |
|                                         | **[模块名2]**                               |
|                                         | 1. **[条目类型]**：[具体说明]                |
|                                         | 2. **关键交互**：「按钮名」→ [跳转目标]      |
```

**右列写法要点**：

1. **模块标题**：粗体 + 变更标注（不带编号，纯作为分节）
2. **条目列表（强制 numbered list）**：每个模块下用 1./2./3. 数字编号，开头**粗体条目类型**冒号引出说明。散段落形式禁止使用
3. **条目类型词表**：校验逻辑 / 校验通过 / 校验失败 / 输入规则 / 展示规则 / 跳转行为 / 空态 / 异常态 / 关键交互 / 对标说明 / 数据来源 / 埋点
4. **复杂条目允许二级缩进**（不新起编号，用 `-` bullet 展开）
5. **状态/枚举**：斜杠分隔
6. **跳转/引用**：用 `→ 见 [编号]`，确保编号存在

**截图占位符格式**：

- Web 端：`🖥 [编号] · [场景名] ← 此处粘贴原型截图`
- Mobile 端：`📱 [编号] · [场景名] ← 此处粘贴原型截图`
- 后台：`⚙️ [编号] · [场景名] ← 此处粘贴原型截图`
- 多端：`📱🖥 [编号] · [场景名] ← 此处粘贴原型截图`

### 第 7 章 · 非功能性需求

```
| 指标           | 要求                            |
|----------------|--------------------------------|
| 首屏加载       | ≤ 2s                           |
| 交互响应       | ≤ 300ms                        |

### 第 8 章 · 埋点与监控

#### 8.0 是否需要埋点（先判断）

满足以下任一条件才写埋点需求，否则写「本需求不涉及自定义埋点」：

| 条件 | 说明 |
|------|------|
| 新页面 / 新核心功能上线 | 需追踪 PV/UV、转化率 |
| 关键用户行为节点 | 报名 / 领取 / 提交等影响业务决策的操作 |
| 重要模块曝光 | 新卡片 / Banner / 弹窗等需衡量触达 |
| 多入口功能 | 需区分流量来源 |
| A/B 实验或活动效果验证 | 需对比不同方案的行为数据 |

**不需要埋点**：纯展示性变更（文案 / 样式）、已有全埋点可覆盖、后端已有记录无需分析行为路径。

#### 8.1 事件类型

| 类型 | 命名后缀 | 触发时机 |
|------|---------|---------|
| 页面浏览 | `_view` | 页面渲染完成时 |
| 模块曝光 | `_show` | 组件在屏幕内可见时 |
| 点击操作 | `_click` | 用户点击时立即上报 |
| 结果上报 | `_result` | 后端返回结果后上报（需带 result 属性） |

#### 8.2 命名规范

格式：`{场景}_{模块}_{动作}`，全小写 + 下划线，≤ 4 个单词。前缀 = 模块缩写。App / Web 同功能用同一事件名，通过 `platform` 属性区分。

#### 8.3 事件列表

每个事件按三要素描述：

```
### [事件展示名]
- **事件名**：`event_name_here`
- **触发时机**：[一句话]
- **自定义属性**：

| 属性名 | 含义 | 类型 | 属性值说明 |
|--------|------|------|-----------|
| activity_id | 活动 ID | string | 透传接口值 |
| entry_source | 入口来源 | string | home=首页 / list=列表页 |
| result | 操作结果 | string | success / fail（结果类事件必填） |
```

属性规则：

- 只写自定义属性（公共属性系统自动上报）
- 单个事件自定义属性 ≤ 5 个
- 多入口功能必写 `entry_source`
- 操作成功 / 失败事件必写 `result`

#### 8.4 报警规则

```
| 指标 | 阈值 | 动作 |
|------|------|------|
| [指标] | > [值] | 容量预警 |
```

## 模板范例 · 4 大要素「❌ 反面 vs ✅ 正面」

每个要素给「容易写成的反面」+「顶尖大厂的正面」对比，PM 直接抄作业。

> 示范背景：交易所「活动中心 V2」迭代 —— 把分散的活动入口聚合到首页 Tab，做活动卡片 + 状态筛选 + 个人活动列表。

### 1.2 北极星 / 配套 / 反向 / 非目标

#### ❌ 反面 · 只写正向、堆模糊指标

```
- 提升活动中心使用率
- 优化用户体验
- 增加用户粘性
- KPI：DAU 增长、留存提升
```

为什么不行：① 没现状值 → 评审看不出难度 ② 没目标值 → 上线后没法判断成败 ③ 没反向指标 → 砍掉首屏其他模块换 DAU 算赢吗？

#### ✅ 正面 · 4 类指标齐全、每个数字都可验证

```
**北极星指标**
- 活动中心 DAU：12K → 25K（3 个月内，灰度全量后稳态值）

**配套指标**（2-3 个）
- 首页活动 Tab CTR ≥ 8%（现状 5.2%）
- 活动详情页转化率（点击 → 报名）≥ 15%
- 我的活动留存（D7）≥ 40%

**反向指标（Guardrail · 不能恶化）**
- 首页 Crash 率 < 0.3‰（现状 0.18‰，红线）
- 客诉率 ≤ 现状 +10%（现状 4.2/万 DAU）
- 整站 D1 留存不下降（现状 78%）
- 活动页加载 P95 ≤ 1.5s（现状 1.1s）

**非目标**
- 不优化加载性能（现状 SLA 已满足）
- 不改 KYC 流程（法规审查窗口未开）
- 不做 Web 端（本期 App-only，Web 排在 P1）
```

### 1.5 Non-goals

#### ❌ 反面 · 留空 / 写「无」一笔带过

```
1.5 Non-goals
（无）
```

为什么不行：评审一眼看出 PM 没想清楚边界。「无」三种可能：① 真没想 ② 想了但不敢承诺 ③ 想了但藏着 —— 三种都坏。

#### ✅ 正面 · 列具体范围 + 每条说原因

```
| 范围 | 原因 |
|------|------|
| 不做活动详情页改版 | 现版本 NPS 78，无优化优先级；如有改版需求另立 PRD |
| 不优化首屏加载性能 | 现状 1.1s 满足 SLA，性能优化排在 V3（H2） |
| 不做 Web 端 | 本期资源仅覆盖 App，Web 排在 P1，预计 V2.1 启动 |
| 不改 KYC 校验逻辑 | 法规审查窗口未开，强行改风险高于收益 |

## docx 生成注意事项

1. **两列表格**是 PRD 的核心格式（Scene 详细说明），左列窄（~35%）右列宽（~65%）
2. **表头底色**用 Anthropic Dark `#141413`，正文表格用浅灰边框
3. **Heading 层级**：H1 = 章 / H2 = 节 / H3 = 子节，确保 TOC 可生成

5. **元信息表**和**非功能性需求表**用等宽两列
6. **场景地图表**用 4 列（编号 / 场景 / 模块 / 优先级）
7. **埋点表**用 3 列（事件 / 触发 / 参数），事件名用 mono 字体
8. **页面层级表**路由列用 mono 字体

详细样式由 `gen_prd_base.py` 的 `C` / `FONT` 字典 + `scene_table` / `make_table` 等函数固化，**不在本文档单独维护**（避免文档代码双轨漂移）。
