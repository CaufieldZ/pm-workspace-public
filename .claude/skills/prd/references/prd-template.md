## 第1章 · 项目背景与目标

### 1.1 背景

[2-3 句话描述现状痛点和做这个项目的原因]

示例：
> 现有 XX 体系分散在多个业务线，缺乏统一的前端聚合展示和后台管理能力。原有入口功能已不能满足业务增长需求。

### 1.2 核心目标

用要点列出 3-4 个核心目标，每个目标带量化指标：

```
- 提升 [指标A]：[手段]，[量化目标]
- 促进 [指标B]：[手段]，[量化目标]
- 降低 [指标C]：[手段]，[量化目标]
- KPI：[关键指标1] > [值]，[关键指标2] > [值]
```

### 1.3 变更范围（迭代项目必填，新项目可省略）

描述本次 PRD 相对**当前线上版本**的差异，不是 PRD 版本之间的 diff。PM 内部多轮迭代痕迹不入此章。

首行声明基线：

```
基线：V[x.x]（YYYY-MM-DD 上线）
```

基线选取规则：第 N 期迭代以「第 N-1 期已发布线上版本」为基线，不以最初版本为基线，避免 delta 越拖越长。

用要点列出每条变更，格式：
```
- [模块/组件] [变更动作]（从现状 xxx 改为 xxx）
- [模块/组件] [砍掉/新增]，[说明]
```

### 1.4 用户角色

```
| 角色     | 端      | 描述               | 核心操作                     |
|----------|---------|--------------------|-----------------------------|
| 普通用户 | Web/App | 浏览、使用功能      | 浏览列表、点击操作、查看结果 |
| 运营人员 | 后台Web | 管理配置            | 上下线、编辑、配置规则       |
| 业务系统 | 服务端  | 自动同步数据        | 推送数据到前端系统           |
```

## 第3/4/5章 · 各 View 详细需求

章标题：「X. [View名称] 详细需求（View N）」

**章首必须加用户故事引言（pm-workflow.md 强制）**：紧跟 h1 调用 `chapter_story(doc, "...")`，一句话 ≤ 30 字，告诉读者这章在讲谁、做什么、为什么。例：
- `chapter_story(doc, "用户进入个人主页查看战绩、收益与动态，是个人空间的核心承载。")`
- `chapter_story(doc, "运营在 CMS 配置活动卡片与白名单，决定前台用户看到的内容。")`

紧接着可保留排版斜体说明：*左侧为原型截图占位，右侧为功能需求说明。*

第 1 章背景目标 / 第 6 章业务规则 / 第 7 章非功能 / 第 8 章埋点 / 第 9 章排期为技术骨架章，免除此要求。

### 两种 Scene 展开模式

**模式 A · scene_table（2 列,默认）**：用于**有独立 UI / 可截图**的 Scene（APP / Web / 有页面改动的后台）。左列原型截图 + 右列 numbered list 说明。

**模式 B · body 级 H3 + numbered list（无截图降级）**：用于**CMS / 纯后台 / 本次 UI 不改只改后端 / 规则说明**等无独立截图的 Scene。不要用 2 列表格给空截图留占位符,直接:

```
## 7.X Scene F-N · [场景名]
（可选）**入口**：[一句话描述]

### [小节标题 1]
1. **条目类型**：内容
2. **条目类型**：内容

### [小节标题 2]
1. **条目类型**：内容
```

**判断准则**：Scene 改动范围内没有 UI 可以截图 → 用模式 B;有 UI 改动(哪怕只改 1 个按钮) → 用模式 A 配截图。本次迭代 UI 不改但要讲历史现状的场景(如 CMS「保留现状 + 后端改动」)也用模式 B。

**代码实现**：模式 B 用 `insert_scene_blocks(anchor, blocks, heading_level=3)`(来自 `update_prd_base`),视觉与 scene_table 右列对齐(numbered list 自动编号 + 内联 `**bold**` 标签解析 + 两空格加 dash 前缀作二级缩进)。参考 patch:`projects/htx-live-streaming-updateQ2/scripts/patch_prd_v27_ch7.py`。

### 模式 A · Scene 详细说明模板(两列表格)

每个 Scene 用一个**两列表格**：

```
| [截图占位]                              | [功能需求]                                  |
|-----------------------------------------|---------------------------------------------|
| 🖥 A · [场景名] ← 此处粘贴原型截图      | **[模块名1]（变更/新增）**                   |
|                                         | 1. **[条目类型]**：[具体说明]                |
|                                         | 2. **[条目类型]**：[具体说明]                |
|                                         | 3. **[条目类型]**：[具体说明]                |
|                                         | **[模块名2]**                               |
|                                         | 1. **[条目类型]**：[具体说明]                |
|                                         | 2. **关键交互**：「按钮名」→ [跳转目标]      |
```

**右列写法要点**：

1. **模块标题**：粗体 + 变更标注（不带编号，纯作为分节）
   ```
   **Header 氛围区（变更：砍数据面板）**
   **Tab 状态筛选**
   **合格投资者校验（R1/R2）**
   ```

2. **条目列表（强制 numbered list）**：每个模块下用 1./2./3. 数字编号，开头**粗体条目类型**冒号引出说明。散段落形式禁止使用（视觉层次弱，与截图标注无法对应）
   ```
   1. **校验逻辑**：进入认购流程前后端双重校验（QI 认证状态 + C 级 ≥ R 级）
   2. **校验通过**：页面顶部展示绿色「合格投资者已认证·C3 匹配 R3」Badge
   3. **校验失败**：拦截进入认购，展示原因 + 引导「去认证」入口
   ```

   **条目类型词表**（常用，不穷举）：校验逻辑 / 校验通过 / 校验失败 / 输入规则 / 展示规则 / 跳转行为 / 空态 / 异常态 / 关键交互 / 对标说明 / 数据来源 / 埋点。

3. **复杂条目允许二级缩进**（不新起编号，用 `-` bullet 展开）：
   ```
   1. **输入规则**：认购金额输入框
      - 最低 100 万元，< 100 万时输入框下方红色提示
      - 实时千分位格式化显示
      - 「下一步」按钮不达标置灰
   ```

3. **状态/枚举**：斜杠分隔
   ```
   进行中（默认）/ 即将开始 / 已结束
   ```

4. **卡片三态**：列出每种状态的视觉和交互
   ```
   进行中（绿）→ CTA「查看」蓝色实心按钮
   即将开始（橙）→ 显示「距开始：xxD xxH」倒计时
   已结束（灰）→ 半透明 + 禁用按钮
   ```

5. **跳转/引用**：
   ```
   「首页」→ 产品主页。
   「专区入口」→ 专区独立 View（跨 Tab）。
   → 见 M-3 专题管理
   ```

6. **砍掉的功能**（如有变更）：
   ```
   砍掉原方案的「xxx + xxx」。
   砍掉「xxx」状态，仅保留「xxx」「xxx」。
   ```

7. **对标说明**（如有）：
   ```
   **对标 [竞品名] [功能名]**
   [竞品] [做法描述]，[我们的做法]。
   ```

**截图占位符格式**：
- Web 端：`🖥 [编号] · [场景名] ← 此处粘贴原型截图`
- Mobile 端：`📱 [编号] · [场景名] ← 此处粘贴原型截图`
- 后台：`⚙️ [编号] · [场景名] ← 此处粘贴原型截图`
- 多端：`📱🖥 [编号] · [场景名] ← 此处粘贴原型截图`

## 第8章 · 埋点与监控

### 8.0 是否需要埋点（先判断）

满足以下任一条件才写埋点需求，否则写「本需求不涉及自定义埋点」：

| 条件 | 说明 |
|------|------|
| 新页面/新核心功能上线 | 需追踪 PV/UV、转化率 |
| 关键用户行为节点 | 报名、领取、提交等影响业务决策的操作 |
| 重要模块曝光 | 新卡片、Banner、弹窗等需衡量触达效果 |
| 多入口功能 | 需区分流量来源 |
| A/B 实验或活动效果验证 | 需对比不同方案的行为数据 |

**不需要埋点**：纯展示性变更（文案/样式）、已有全埋点可覆盖且无需细分属性、后端已有记录且无需分析行为路径。

### 8.1 事件类型

| 类型 | 命名后缀 | 触发时机 |
|------|---------|---------|
| 页面浏览 | `_view` | 页面渲染完成时 |
| 模块曝光 | `_show` | 组件在屏幕内可见时 |
| 点击操作 | `_click` | 用户点击时立即上报 |
| 结果上报 | `_result` | 后端返回结果后上报（需带 result 属性） |

### 8.2 命名规范

格式：`{场景}_{模块}_{动作}`，全小写+下划线，≤ 4 个单词。

- 前缀 = 模块缩写（如 `ac_` = activity center, `tm_` = trade mining）
- App/Web 同功能用同一事件名，通过 `platform` 属性区分
- 后台操作 = `mgt_[动作]`

### 8.3 事件列表

每个事件按三要素描述：

```
### [事件展示名]
- **事件名**：`event_name_here`
- **触发时机**：[一句话，如「用户点击报名按钮时」]
- **自定义属性**：

| 属性名 | 含义 | 类型 | 属性值说明 |
|--------|------|------|-----------|
| activity_id | 活动 ID | string | 透传接口值 |
| entry_source | 入口来源 | string | home=首页 / list=列表页 |
| result | 操作结果 | string | success / fail（结果类事件必填） |
```

**属性规则**：
- 只写自定义属性（用户 ID/设备 ID/时间/版本/AB 分组等公共属性系统自动上报）
- 单个事件自定义属性 ≤ 5 个
- 多入口功能必写 `entry_source`
- 操作成功/失败事件必写 `result`

### 8.4 报警规则

```
| 指标 | 阈值 | 动作 |
|------|------|------|
| [指标] | > [值] | 容量预警 |
| [失败率] | > [值] | 报警 |
| [审计] | — | 全量操作日志 |
```

## docx 生成注意事项

1. **两列表格**是 PRD 的核心格式（Scene 详细说明），左列窄（~35%）右列宽（~65%）
2. **表头底色**用浅蓝（`D5E8F0`），正文表格用浅灰边框
3. **Heading 层级**：H1=章 / H2=节 / H3=子节，确保 TOC 可生成
4. **粗体**用于模块标题、关键字段名、变更标注
5. **元信息表**和**非功能性需求表**用等宽两列
6. **场景地图表**用 4 列（编号/场景/模块/优先级）
7. **埋点表**用 3 列（事件/触发/参数），事件名用等宽字体
8. **页面层级表**路由列用等宽字体
